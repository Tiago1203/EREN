# ADR-3050: Immutable Evidence Tree Construction (Option 4)

## Estado

✅ Accepted - 2026-07-22

---

## Contexto

Durante la auditoría arquitectónica de FASE 3, se identificó que los tests de `EvidenceTreeBuilder` fallaban con `FrozenInstanceError` porque el código intentaba modificar dataclasses declarados con `frozen=True`.

**Problema Específico:**

```python
@dataclass(frozen=True)
class TreeNode:
    node_id: str
    children: list  # Campo intentando modificarse post-creación

# Código que ROMPÍA frozen:
supporting_node.children.append(child)  # ❌ FrozenInstanceError
root.children = [supporting_node]       # ❌ FrozenInstanceError
```

---

## Decisión

Se seleccionó la **Opción 4** para resolver el problema:

> **Crear TreeNodes con `children` desde el inicio**

En lugar de crear nodos vacíos y luego agregar children con `.append()`, se construyen con toda su lista de children de una vez.

### Implementación

```python
# ❌ ANTES (rompe frozen, 3 líneas problemáticas):
supporting_node = TreeNode(node_id="supporting", children=[])
supporting_node.children.append(child)  # ← Error
root.children = [supporting_node]       # ← Error

# ✅ DESPUÉS (Opción 4 - inmutable):
supporting_children = [
    TreeNode(node_id=f"supp_{i}", label=..., children=[])
    for i, e in enumerate(supporting[:5])
]
supporting_node = TreeNode(
    node_id="supporting",
    children=supporting_children,  # ← Creado con children
)
root = TreeNode(
    node_id="root",
    children=[supporting_node, contradicting_node],  # ← Con todo
)
```

---

## Opciones Consideradas

| Opción | Descripción | Resultado |
|--------|-------------|-----------|
| **Opción 1** | Quitar `frozen=True` | ❌ Pierde inmutabilidad, rompe auditoría |
| **Opción 2** | Usar `object.__setattr__` | ❌ "Hack", poco legible |
| **Opción 3** | Crear nuevas instancias | ⚠️ Funciona pero más objetos intermedios |
| **Opción 4** | Crear con children desde el inicio | ✅ **SELECCIONADA** |

---

## Opciones Evaluadas

### Por qué Opción 4 sobre Opción 3

| Criterio | Opción 3 | Opción 4 | Ganadora |
|----------|----------|----------|----------|
| **Rendimiento** | Más objetos | Menos objetos | Opción 4 |
| **Legibilidad** | Imperativo | Declarativo | Opción 4 |
| **Mantenibilidad** | Difícil seguir | Flujo claro | Opción 4 |
| **Testing** | Complejo | Simple | Opción 4 |

---

## Consecuencias

### Positivas

1. ✅ **Inmutabilidad preservada** - `frozen=True` funciona como diseñado
2. ✅ **Código declarativo** - Más legible y mantenible
3. ✅ **Rendimiento mejorado** - Menos objetos intermedios creados
4. ✅ **Seguro para concurrencia** - Sin shared state mutable
5. ✅ **Auditoría clínica** - Trazabilidad garantizada
6. ✅ **Concatenación de fases** - Código predecible para FASE 4+

### Negativas

1. ⚠️ **Refactor requerido** - Cambio en el método `build()`
2. ⚠️ **Lists comprehensions** - Requiere familiaridad con Python

**Cambios:**

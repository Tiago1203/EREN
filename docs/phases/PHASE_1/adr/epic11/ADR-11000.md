# ADR-11000: Domain AI Integration Architecture

**Status:** Proposed  
**Date:** 2026-07-20  
**Deciders:** Architecture Board  

---

## Context

EREN FASE 1 implementó un dominio de negocio completo con 11 bounded contexts (Device, Incident, Knowledge, Recommendation, WorkOrder, Capacity, Staffing, Organization, Department, Inventory, Asset). FASE 2 implementó el AI Core con Conversation, Memory, Prompt, Context, Tools, Response, Providers, y Sessions.

**Problema:** El AI Core no tiene una forma desacoplada de acceder al dominio de negocio. Actualmente:
- AI Core accedería directamente a repositorios
- Entidades de dominio estarían expuestas a AI
- No hay separación CQRS (Commands vs Queries)
- No hay DTOs específicos para consumo de AI

---

## Decision

Crear una nueva capa `core/application/services/` que funcione como fachada oficial entre el dominio de negocio y el AI Core.

### Arquitectura de la Capa

```
core/application/
├── services/
│   ├── query/           # Query Services (CQRS Read)
│   ├── command/          # Command Services (CQRS Write)
│   ├── domain/           # Domain Services (AI-Ready)
│   └── events/          # Event Services
├── dto/                 # Data Transfer Objects
└── interfaces/          # Service Interfaces
```

### Principios

1. **Dependency Inversion**: AI Core depende de abstracciones (interfaces), no de implementaciones
2. **CQRS**: Separación clara entre operaciones de lectura (Query) y escritura (Command)
3. **DTOs**: Entidades de dominio nunca se exponen directamente a AI
4. **Single Responsibility**: Cada servicio tiene una responsabilidad clara
5. **Open/Closed**: Nuevos servicios sin modificar existentes

---

## Consequences

### Positive
- AI Core completamente desacoplado del dominio
- Dominio de negocio protegido de cambios en AI Core
- Tests más fáciles de escribir (servicios stubbables)
- Clear boundaries entre capas

### Negative
- Más código (fachadas adicionales)
- Duplicación potencial de lógica de query
- Curva de aprendizaje para nuevos desarrolladores

### Risks
- Mantener sincronización entre DTOs y entidades
- Posible performance overhead por transformación de DTOs

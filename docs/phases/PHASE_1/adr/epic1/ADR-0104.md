# ADR-0104: Unit of Work Pattern

**Status:** Accepted

**Date:** 2026-07-16

**Deciders:** Architecture Board

---

## Context

Multiple bounded contexts may need to persist data in a single transaction. EREN needs a transactional boundary that coordinates all repositories and the outbox pattern atomically.

## Decision

Implement the **Unit of Work pattern** as an async context manager that coordinates repositories and outbox within a single database transaction.

## Rationale

- **Atomicity**: All changes commit together or rollback together
- **Outbox integration**: Events are written to the outbox table in the same transaction
- **Clean API**: `async with UnitOfWork() as uow` pattern
- **Repository access**: All repositories accessed through a single `uow`

## Usage

```python
async with UnitOfWork() as uow:
    device = await uow.devices.save(device)
    incident = await uow.incidents.save(incident)
    uow.outbox().append(
        event_type="device.incident_created",
        payload={"device_id": str(device.id)},
    )
    await uow.commit()
    # Outbox flushed inside commit() → same transaction
    # RabbitMQ publish happens asynchronously via OutboxWorker
```

## Consequences

### Positive

- All-or-nothing commits across bounded contexts
- No partial state if any repository fails
- Outbox events are always persisted with data (no event loss)

### Negative

- Long-running transactions can cause lock contention
- OutboxWorker must run to actually publish events to RabbitMQ

# ADR-0108: OpenTelemetry Observability

**Status:** Accepted

**Date:** 2026-07-16

**Deciders:** Architecture Board

---

## Context

EREN needs distributed tracing across API requests, database queries, and RabbitMQ message handling to diagnose performance issues and failures in production.

## Decision

Use **OpenTelemetry (OTel) 1.24+** with the following instrumentation:

| Component | Instrumentation | Exporter |
|-----------|---------------|----------|
| FastAPI | `opentelemetry-instrumentation-fastapi` | OTLP gRPC |
| SQLAlchemy | `opentelemetry-instrumentation-sqlalchemy` | OTLP gRPC |
| Redis | `opentelemetry-instrumentation-redis` | OTLP gRPC |
| RabbitMQ | Manual spans | OTLP gRPC |
| Custom | `opentelemetry-api` + `opentelemetry-sdk` | OTLP gRPC |

## Rationale

- **Vendor-neutral**: Can switch exporters (Jaeger, Zipkin, Tempo)
- **Auto-instrumentation**: FastAPI and SQLAlchemy require minimal code
- **Correlation IDs**: Links traces from HTTP to DB to RabbitMQ
- **Open source**: No vendor lock-in

## Consequences

### Positive

- Full request tracing from HTTP to DB to RabbitMQ
- Automatic span attributes (HTTP method, status code, SQL query, DB duration)
- Correlates logs with traces

### Negative

- Additional latency from tracing (mitigated by sampling)
- OTel endpoint must be available or tracing disabled gracefully

## Implementation

```python
# setup_instrumentation(app)
FastAPIInstrumentor.instrument_app(app)
SQLAlchemyInstrumentor().instrument()
# Manual spans for RabbitMQ:
with tracer.start_as_current_span("rabbitmq.publish") as span:
    span.set_attribute("messaging.system", "rabbitmq")
    span.set_attribute("messaging.destination", topic)
    await exchange.publish(message, routing_key=topic)
```

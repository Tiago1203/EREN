# ADR-0110: Health Check Strategy

**Status:** Accepted

**Date:** 2026-07-16

**Deciders:** Architecture Board

---

## Context

EREN needs health checks for Kubernetes liveness and readiness probes, Docker health checks, and internal monitoring dashboards.

## Decision

Implement a **four-tier health check system**:

| Endpoint | Purpose | Checks |
|----------|---------|--------|
| `GET /api/v1/health` | Simple liveness | App is running |
| `GET /api/v1/health/live` | Kubernetes liveness | App is alive |
| `GET /api/v1/health/ready` | Kubernetes readiness | DB + Redis healthy |
| `GET /api/v1/health/full` | Full diagnostics | All dependencies |

## Rationale

- **Kubernetes-native**: Matches K8s liveness/readiness probe semantics
- **Selective readiness**: `ready` only returns 200 when DB is available
- **Zero-dependency health**: `health` always works (no external deps)
- **Full diagnostics**: `full` for Grafana/health dashboards

## Implementation

```python
@router.get("/health/ready")
async def readiness() -> ReadinessResponse:
    db_check, redis_check = await asyncio.gather(
        _check_database(),
        _check_redis(),
    )
    ready = db_check.get("status") == "healthy"
    return ReadinessResponse(
        status="ready" if ready else "not_ready",
        checks={"database": db_check, "redis": redis_check},
    )
```

## Docker Health Check

```dockerfile
HEALTHCHECK --interval=30s --timeout=10s --retries=3 \
    CMD curl -f http://localhost:8000/api/v1/health/live || exit 1
```

## Consequences

### Positive

- Kubernetes can safely restart pods when dependencies fail
- Clear visibility into dependency health
- No /health endpoint crashes

### Negative

- Multiple endpoints to monitor
- `ready` endpoint adds slight DB query overhead

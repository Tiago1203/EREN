# ADR-0109: Structured Logging

**Status:** Accepted

**Date:** 2026-07-16

**Deciders:** Architecture Board

---

## Context

EREN needs production-ready logging that is machine-parseable, correlates with traces, and includes tenant context for multi-tenancy debugging.

## Decision

Use **python-json-logger 2.0+** for structured JSON logs in production, with correlation IDs injected via middleware.

## Log Format

```json
{
  "timestamp": "2026-07-16T10:30:00.000Z",
  "level": "INFO",
  "logger": "app.infrastructure.repositories.device",
  "message": "Device saved successfully",
  "request_id": "abc-123",
  "trace_id": "def-456",
  "tenant_id": "hospital-001",
  "device_id": "dev-789",
  "service": "eren-api"
}
```

## Correlation IDs

Every HTTP request gets:
- `request_id`: Unique per request (UUID)
- `trace_id`: OpenTelemetry trace ID
- `tenant_id`: From JWT/inference

All log entries include these fields via `RequestContextMiddleware`.

## Log Levels

| Level | Usage |
|-------|-------|
| ERROR | Exceptions, failed operations, unhandled errors |
| WARN | Retries, circuit breaker open, rate limiting |
| INFO | Request/response, startup, shutdown |
| DEBUG | SQL queries, cache hits/misses, RabbitMQ publishes |

## Consequences

### Positive

- Machine-parseable logs for ELK/Loki/Grafana
- Full request tracing via correlation IDs
- Tenant context in every log entry

### Negative

- JSON logs are harder to read during local development (mitigated with human-readable format in dev)

## Implementation

```python
# CorrelationJsonFormatter (production)
class CorrelationJsonFormatter(jsonlogger.JsonFormatter):
    def add_fields(self, log_record, record, message_dict):
        super().add_fields(log_record, record, message_dict)
        log_record["request_id"] = get_request_context().request_id
        log_record["trace_id"] = get_request_context().request_id
        log_record["tenant_id"] = get_request_context().tenant_id
```

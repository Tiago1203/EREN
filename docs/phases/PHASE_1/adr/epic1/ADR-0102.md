# ADR-0102: Alembic for Database Migrations

**Status:** Accepted

**Date:** 2026-07-16

**Deciders:** Architecture Board

---

## Context

EREN needs a migration tool to evolve the database schema across environments (dev, staging, production) with versioning, rollbacks, and multi-schema support.

## Decision

Use **Alembic 1.13+** with schema-per-bounded-context organization.

## Rationale

- **Industry standard**: Most widely-used SQLAlchemy migration tool
- **Version control**: Every schema change is a numbered revision
- **Autogenerate**: Detects model changes automatically
- **Rollback**: Full downgrade support
- **Multi-schema**: Supports PostgreSQL schemas per bounded context

## Schema Organization

| Schema | Bounded Context |
|--------|-----------------|
| `incident` | Engineering Incident |
| `device` | Device Registry |
| `recommendation` | AI Recommendation |
| `knowledge` | Knowledge Graph |
| `work_order` | Maintenance Work Order |
| `incident.outbox_events` | Outbox pattern |

## Consequences

### Positive

- Every migration is reversible
- CI/CD can run migrations automatically
- Schema history is version-controlled

### Negative

- Migration conflicts possible with parallel development (use rebase workflow)
- Manual review required for autogenerate output

## Implementation

```bash
# Generate migration
cd apps/api
uv run alembic revision --autogenerate -m "add device calibration fields"

# Apply
uv run python scripts/migrate.py upgrade

# Rollback
uv run python scripts/migrate.py downgrade -1
```

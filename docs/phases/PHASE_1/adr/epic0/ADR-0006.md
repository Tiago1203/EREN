# ADR-0006: Qdrant for Semantic Search

**Status:** Accepted

**Date:** 2026-02-15

**Deciders:** Architecture Board

---

## Context

EREN needs semantic search for:
- Medical evidence retrieval for CDS recommendations
- Device documentation search
- Knowledge base queries using natural language

Traditional keyword search (PostgreSQL full-text) cannot handle semantic similarity.

---

## Decision

**Qdrant is used as the vector store for semantic search.**

Embeddings are generated by an LLM and stored in Qdrant. PostgreSQL remains the source of truth for metadata.

---

## Reasons

1. **Semantic similarity:** cosine/inner-product similarity for evidence retrieval
2. **Filtering:** Pre-filter by tenant_id, document type before similarity search
3. **Performance:** Optimized for million-scale vector operations
4. **Deployment:** Can run on Kubernetes (self-hosted) or Qdrant Cloud
5. **Epic 0 alignment:** `EREN_CONSISTENCY_MODEL.md` specifies Qdrant as derived vector store

---

## Collections

| Collection | Dimension | Use Case |
|-----------|-----------|----------|
| `knowledge_base_evidence` | 1536 | Medical evidence embeddings |
| `device_documentation` | 1536 | Device manuals, specifications |
| `clinical_guidelines` | 1536 | Treatment guidelines |
| `alarm_history` | 1536 | Alarm patterns for similarity |

---

## Consequences

### Positive
- Natural language search: "find similar alarms to this one"
- Evidence retrieval for CDS with semantic matching
- Fast approximate nearest neighbor (ANN) search

### Negative
- Additional infrastructure
- Qdrant is derived — must be rebuilt from source data
- Embedding model must be consistent

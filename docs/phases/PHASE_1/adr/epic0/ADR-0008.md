# ADR-0008: Multi-Tenancy Strategy

**Status:** Accepted

**Date:** 2026-03-01

**Deciders:** Architecture Board

---

## Context

EREN serves multiple hospital tenants on a single deployment. Each tenant must be completely isolated from others. PHI data (patient records) must never leak between tenants.

---

## Decision

**Shared database + tenant_id in all tables + Row-Level Security (RLS) + Application enforcement.**

Every table has a `tenant_id` column. PostgreSQL RLS enforces isolation at the database level. The application also enforces isolation at the connection level.

See `EREN_MULTITENANCY_STRATEGY.md` (Epic 0.1) and `ADR-0084` (RLS in Alembic) for full details.

---

## Reasons

1. **Defense in depth:** RLS + application enforcement = no single point of failure
2. **HIPAA compliance:** DB-level isolation prevents accidental cross-tenant access
3. **Cost efficiency:** Shared infrastructure, single deployment to manage
4. **Epic 0 alignment:** Explicitly defined in `EREN_MULTITENANCY_STRATEGY.md`

---

## Enforcement Layers

```
Layer 1: Application    → tenant_id required in every request
Layer 2: Repository     → tenant_id injected in every query
Layer 3: RLS            → PostgreSQL blocks cross-tenant reads
Layer 4: Audit          → Every query logged with tenant_id
```

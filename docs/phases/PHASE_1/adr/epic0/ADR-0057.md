# ADR-0057: LLM Provider Abstraction

**Status:** Accepted

**Date:** 2026-07-16

**Deciders:** Architecture Board

---

## Context

EREN uses LLMs for reasoning and explanation generation. The LLM provider may change over time (OpenAI → Anthropic → Azure OpenAI → local model).

---

## Decision

**LLM providers are abstracted behind a ProviderPort interface. The application never calls an LLM directly.**

```python
class LLMProvider(Protocol):
    async def complete(self, prompt: Prompt, config: LLMConfig) -> LLMResponse: ...
    async def embed(self, text: str) -> Embedding: ...

# Implementations
class OpenAIProvider(LLMProvider): ...
class AnthropicProvider(LLMProvider): ...
class AzureOpenAIProvider(LLMProvider): ...
```

---

## Provider Selection

```python
# Configured via environment
LLM_PROVIDER=openai  # openai | anthropic | azure
```

---

## Fallback Chain

If the primary LLM fails:
1. OpenAI GPT-4 → GPT-3.5
2. GPT-3.5 → Anthropic Claude
3. Claude → Rule-based (no LLM)

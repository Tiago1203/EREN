# ADR-0001: Hexagonal Architecture

**Status:** Accepted

**Date:** 2026-01-15

**Deciders:** Architecture Board

---

## Context

EREN needs an architecture that separates business logic from infrastructure concerns (database, messaging, external APIs). This allows:
- Business logic to be testable without external dependencies
- Easy swapping of infrastructure components (e.g., PostgreSQL → MySQL)
- Clear boundaries between bounded contexts
- Independent deployment of components

---

## Decision

**EREN adopts Hexagonal Architecture (Ports and Adapters).**

The application is structured as:
- **Domain Core:** Pure business logic, no framework dependencies
- **Ports:** Interfaces (abstract classes) that define how the domain interacts with the outside world
- **Adapters:** Implementations of ports (database repositories, message publishers, API clients)

```
┌─────────────────────────────────────────────────────────┐
│                      Application Layer                    │
│  ┌───────────────────────────────────────────────────┐  │
│  │                    Domain Core                      │  │
│  │   Entities │ Value Objects │ Aggregates │ Services  │  │
│  │                      Ports (Interfaces)            │  │
│  └───────────────────────────────────────────────────┘  │
│                         │                               │
│   ┌─────────────────────┼──────────────────────────┐  │
│   │                     │                              │  │
│   ▼                     ▼                              ▼  │
│ ┌────────┐      ┌─────────────┐      ┌─────────────┐  │
│ │ REST   │      │ PostgreSQL  │      │ Kafka       │  │
│ │ Adapter │      │ Adapter     │      │ Adapter      │  │
│ └────────┘      └─────────────┘      └─────────────┘  │
│ ┌────────┐      ┌─────────────┐      ┌─────────────┐  │
│ │ Celery │      │ Neo4j       │      │ LLM        │  │
│ │ Adapter │      │ Adapter     │      │ Adapter     │  │
│ └────────┘      └─────────────┘      └─────────────┘  │
└─────────────────────────────────────────────────────────┘
```

---

## Reasons

1. **Testability:** Domain logic tested with in-memory adapters, no real DB/messaging needed
2. **Swappable infrastructure:** Change from PostgreSQL to MySQL without touching domain
3. **Clear bounded context boundaries:** Each context has its own ports and adapters
4. **Epic 0 alignment:** `EREN_ARCHITECTURE_BLUEPRINT.md` specifies hexagonal architecture
5. **Industry proven:** Used by healthcare, fintech companies with strict compliance needs

---

## Consequences

### Positive
- Domain logic is framework-agnostic
- Easy to test business rules in isolation
- Clear dependency direction: infrastructure depends on domain, not the other way around
- Supports multiple adapters for the same port (e.g., production vs test database)

### Negative
- More initial structure and abstraction layers
- Requires discipline to keep domain pure (no framework imports)
- Slightly more code for the adapter layer

## Implementation

```python
# Domain Core (no framework imports)
class Incident(BaseEntity):
    """Domain entity — pure business logic."""
    def escalate(self) -> None:
        if self.priority < Priority.HIGH:
            self.priority = Priority.HIGH
            self.escalation_level += 1

# Port (interface)
class IncidentRepository(Protocol):
    """Port — defines how domain interacts with persistence."""
    async def save(self, incident: Incident) -> Incident: ...
    async def get_by_id(self, id: UUID) -> Incident | None: ...

# Adapter (implementation)
class PostgresIncidentRepository:
    """Adapter — implements port using PostgreSQL."""

        ...
```

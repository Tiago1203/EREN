# ADR-0003: Event-Driven Architecture

**Status:** Accepted

**Date:** 2026-02-01

**Deciders:** Architecture Board

---

## Context

EREN needs to communicate state changes between bounded contexts. When an incident is created, the alarm context needs to know. When a device fails, the CDS context needs to generate recommendations. Traditional synchronous calls create tight coupling.

---

## Decision

**EREN uses Event-Driven Architecture for inter-context communication.**

State changes publish domain events to a message broker. Other contexts subscribe and react asynchronously. This decouples bounded contexts and enables scalability.

See `EREN_EVENT_ARCHITECTURE.md` (Epic 0.5) and `ADR-0081` (Kafka) for full implementation details.

---

## Reasons

1. **Decoupling:** Bounded contexts evolve independently
2. **Scalability:** Consumers scale independently from producers
3. **Resilience:** Events are persisted — consumers can replay
4. **Epic 0 alignment:** `EREN_EVENT_ARCHITECTURE.md` is a core Epic 0.5 document
5. **Audit trail:** All events are stored, enabling compliance

---

## Event Delivery Guarantees

| Event Type | Guarantee |
|-----------|-----------|
| Clinical events | At-least-once |
| Biomedical events | At-least-once |
| Hospital events | At-least-once |
| Audit events | Exactly-once |
| Critical alarms | At-least-once + SMS fallback |

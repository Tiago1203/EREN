# ADR-0204: WorkOrder in Incident Bounded Context

**Status:** Proposed

**Date:** 2026-07-16

**Deciders:** Architecture Board

---

## Context

Work Order is a domain concept in EPIC 2. Currently it only exists in `apps/api/app/domain/work_order/` — not in `core/`. This violates ADR-0200 (core/ is the only source of truth for domain logic).

Work Order represents maintenance work generated from incidents. It belongs to the Incident bounded context.

---

## Decision

**WorkOrder is a sub-aggregate of the Incident bounded context, owned by `core/incident/`.**

```
core/incident/
├── domain/
│   ├── entities/
│   │   ├── incident.py        ← EngineeringIncident (Aggregate Root)
│   │   └── work_order.py       ← WorkOrder (Sub-aggregate) [PENDING MOVE]
│   ├── repositories/
│   │   ├── incident_repository.py
│   │   └── work_order_repository.py
│   └── services/
│       ├── incident_service.py
│       └── work_order_service.py
└── value_objects/
    └── work_order_status.py    ← WorkOrderStatus, WorkOrderType
```

**WorkOrder as sub-aggregate (not separate aggregate):**
- WorkOrder is meaningless without an Incident
- Changes to WorkOrder must respect Incident invariants
- Single transaction for incident + work order changes
- Easier consistency model

**WorkOrder aggregate (sub):**
- Identity: WorkOrderId, derived from IncidentId: `wo_{incident_id}_{sequence}`
- State: WorkOrderStatus, WorkOrderType, assigned_to, scheduled_date
- Lifecycle: CREATED → ASSIGNED → SCHEDULED → IN_PROGRESS → COMPLETED / CANCELLED

**Invariants:**
- Work order must reference an incident
- Cannot complete work order without completing incident investigation
- Only assigned technician can update status

---

## Current State

WorkOrder currently exists only in `apps/api/app/domain/work_order/`. It must be moved to `core/incident/domain/`.

The `apps/api/app/domain/work_order/` service and repository are orchestration — they should remain in `apps/api/` but call `core/incident/domain/work_order.py`.

---

## Consequences

**Positive:**
- Work order business logic in core/ — framework-independent
- Enforces architectural rule (ADR-0200)
- Easier to test without FastAPI
- Consistent with Investigation sub-aggregate pattern

**Negative:**
- Refactoring required: move files from `apps/api/app/domain/work_order/` to `core/incident/domain/`
- May need to update repository implementations
- Migration of existing work orders

**Action required:**
1. Create `core/incident/domain/entities/work_order.py`
2. Create `core/incident/domain/value_objects/work_order_status.py`
3. Create `core/incident/domain/repositories/work_order_repository.py`
4. Create `core/incident/domain/services/work_order_service.py`
5. Refactor `apps/api/app/domain/work_order/service.py` to call `core/` (orchestration only)
6. Delete duplicate logic in `apps/api/app/domain/work_order/` — keep only orchestration

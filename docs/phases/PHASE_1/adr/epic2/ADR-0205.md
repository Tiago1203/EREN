# ADR-0205: Knowledge Domain Isolation

**Status:** Proposed

**Date:** 2026-07-16

**Deciders:** Architecture Board

---

## Context

The `core/knowledge/` directory contains both domain logic AND infrastructure/engines:

```
core/knowledge/
├── domain/                    ← Domain (framework-independent)
│   ├── entities/
│   ├── repositories/
│   ├── services/
│   └── value_objects/
├── engine.py               ← INFRASTRUCTURE (uses embeddings, vector DB)
├── knowledge_engine.py      ← INFRASTRUCTURE (RAG pipeline)
├── knowledge_router.py      ← INFRASTRUCTURE (FastAPI routing)
├── knowledge_registry.py    ← INFRASTRUCTURE
├── knowledge_metrics.py     ← INFRASTRUCTURE
├── embeddings/             ← INFRASTRUCTURE (vector embeddings)
├── rag/                    ← INFRASTRUCTURE (RAG pipeline)
└── retrieval/              ← INFRASTRUCTURE
```

This violates ADR-0200: `core/` must be framework-independent. Embeddings and vector DB clients are infrastructure.

---

## Decision

**Move all infrastructure/engine code out of `core/knowledge/` into `apps/api/` or a new `core/knowledge_engine/` package.**

```
core/knowledge/                   (framework-independent domain)
└── domain/                       ← ONLY this stays in core/knowledge/
    ├── entities/
    ├── repositories/
    ├── services/
    └── value_objects/

apps/api/app/knowledge_engine/    (infrastructure — FastAPI, embeddings, Qdrant)
├── engine.py                    ← embeds Qdrant client
├── rag_pipeline.py              ← RAG pipeline with embeddings
├── retrieval/                   ← retrieval strategies
└── knowledge_router.py         ← FastAPI routes

OR:

core/knowledge_engine/            (infrastructure but not apps/api/)
└── engine.py                   ← embeddings, vector DB
├── rag_pipeline.py
└── retrieval/
```

**Criteria for what stays in `core/knowledge/domain/`:**
- Pure Python dataclasses (entities, value objects)
- Abstract repository interfaces
- Domain services with NO external dependencies
- Domain events (frozen dataclasses)
- Business rules and invariants

**What must move out:**
- `core/knowledge/engine.py` — uses embeddings
- `core/knowledge/knowledge_engine.py` — uses Qdrant
- `core/knowledge/knowledge_router.py` — uses FastAPI
- `core/knowledge/embeddings/` — infrastructure
- `core/knowledge/rag/` — infrastructure
- `core/knowledge/retrieval/` — infrastructure

**Domain contract:** The `Knowledge` repository interface stays in `core/knowledge/domain/`. The implementation (Qdrant, embeddings) lives in infrastructure.

---

## Consequences

**Positive:**
- `core/knowledge/domain/` is fully framework-independent
- Embeddings and vector DB can evolve without touching domain
- Domain tests run without any infrastructure
- Clear separation: domain tests are unit tests, engine tests are integration tests

**Negative:**
- Refactoring required: move ~15 files
- May break existing imports
- Need to update test structure

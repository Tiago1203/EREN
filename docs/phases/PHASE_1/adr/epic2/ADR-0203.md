# ADR-0203: Context Integration Pattern

**Status:** Accepted

**Date:** 2026-07-16

**Deciders:** Architecture Board

---

## Context

How do bounded contexts communicate? Direct synchronous calls (HTTP, gRPC) create tight coupling and distributed transactions. We need a pattern that maintains autonomy while enabling rich integration.

---

**Why RabbitMQ + Outbox:**
- RabbitMQ provides reliable message delivery
- Outbox pattern ensures atomicity: entity save + event publish in one DB transaction
- DLQ (Dead Letter Queue) for failed messages
- Idempotent consumers (every event has unique event_id)

**Event flow:**
1. Context A saves entity вҶ’ Outbox table (same transaction)
2. Outbox worker reads pending events вҶ’ publishes to RabbitMQ
3. Context B consumes from RabbitMQ вҶ’ updates own state
4. On failure вҶ’ DLQ вҶ’ retry with backoff

**No direct calls between contexts:**
- Device cannot call Incident directly
- Incident cannot call Recommendation directly
- Knowledge cannot be queried synchronously by other contexts

**Cross-context reads:**
- Use PostgreSQL materialized views or read replicas
- Or use a shared reporting database (read-only copies)
- Do NOT use direct DB access across context boundaries

## Integration Events

| Publisher | Event | Subscribers |
|-----------|-------|-----------|
| Device Context | `DeviceStatusChanged` | Incident Context, Alarm Context |
| Incident Context | `IncidentReported` | Recommendation Context, Knowledge Context |
| Incident Context | `IncidentResolved` | Recommendation Context, Knowledge Context |
| Recommendation Context | `RecommendationCreated` | Incident Context |
| Recommendation Context | `RecommendationFeedbackReceived` | Knowledge Context |

---

## Consequences

**Positive:**
- Contexts remain autonomous
- No distributed transactions
- Events are a permanent record of what happened
- Easy to add new consumers
- Resilience: consumers can be down without blocking publishers

**Negative:**
- Eventual consistency (not immediate)
- More complex debugging (tracing across contexts)
- Events can get out of order вҖ” consumers must handle this
- Schema evolution requires versioning strategy

# ADR-0200: Domain Model Architecture

**Status:** Accepted

**Date:** 2026-07-16

**Deciders:** Architecture Board

---

## Context

EREN has two parallel domain code locations:
1. `core/` — Hexagonal architecture domain layer
2. `apps/api/app/domain/` — FastAPI application layer

Both contained entities, repositories, and services. This creates ambiguity about where business logic lives and violates the principle that domain must be framework-independent.

---

## Decision

**`core/` is the ONLY source of truth for domain logic.**

```
apps/api/ (application/infrastructure)
  ├── routers/         FastAPI endpoints — no business logic
  ├── middleware/       Auth, audit, request context
  ├── tasks/           Celery tasks — orchestration only
  └── infrastructure/
      ├── models/      SQLAlchemy ORM models — infrastructure
      ├── repositories/ Repository implementations — infrastructure
      ├── messaging/   RabbitMQ, Redis, Outbox — infrastructure
      └── observability/ Logging, tracing — infrastructure
  └── domain/          THIN orchestration only — no business rules here

core/ (domain — framework-independent)
  ├── device/          Bounded context — pure Python, no FastAPI, no SQLAlchemy
  ├── incident/        Bounded context — pure Python
  ├── recommendation/ Bounded context — pure Python
  ├── knowledge/       Bounded context — pure Python
  └── shared/          Shared Kernel — pure Python
```

**Rules:**
- `core/` must not import from `apps/api/`
- `core/` must not use FastAPI, SQLAlchemy, Redis, or RabbitMQ
- `core/` must use only stdlib + dataclasses + typing
- `apps/api/app/domain/` must only orchestrate — call `core/` services, never duplicate logic
- Repository implementations live in `apps/api/app/infrastructure/repositories/`, not in `core/`

---

## Consequences

**Positive:**
- Domain logic is portable and testable without FastAPI
- Clear separation of concerns
- Business rules cannot be entangled with HTTP framework
- Easier to evolve domain without breaking infrastructure

**Negative:**
- Refactoring needed: move WorkOrder from `apps/api/app/domain/` to `core/incident/`
- Refactoring needed: remove `apps/api/app/domain/device/` (already exists in `core/device/`)
- Refactoring needed: remove `apps/api/app/domain/patient/` and `apps/api/app/domain/diagnosis/` (not EPIC 2 scope)

# ADR-0202: Incident Sub-Aggregate (Investigation)

**Status:** Accepted

**Date:** 2026-07-16

**Deciders:** Architecture Board

---

## Context

The Incident aggregate has grown to include investigation tracking. Should investigation be a separate aggregate or a sub-aggregate of EngineeringIncident?

A separate Investigation aggregate would require:
- Maintaining consistency between Incident and Investigation across contexts
- ID references and foreign keys
- Two-phase commits

A sub-aggregate is owned by the parent aggregate and lives within the same consistency boundary.

---

## Decision

**Investigation is a sub-aggregate of EngineeringIncident, not a separate aggregate.**

```
EngineeringIncident (Aggregate Root)
└── Investigation (Sub-aggregate, owned)
    ├── InvestigationId
    ├── Actions taken
    ├── Evidence collected
    └── Notes
```

**Why sub-aggregate over separate aggregate:**
- Investigation is meaningless without an incident
- Changes to investigation must respect incident invariants (cannot add actions to closed incidents)
- Both live in the same consistency boundary
- No two-phase commit needed
- Same persistence transaction

**Rules:**
- InvestigationId is generated from parent IncidentId: `inv_{uuid}`
- Investigation actions are added to the parent's pending event list
- Closed incidents reject investigation modifications (invariant)
- Investigation data is serialized alongside Incident in a single table or as JSONB

---

## Consequences

**Positive:**
- Single transaction for incident + investigation changes
- Invariants enforced in one place
- Simpler consistency model

**Negative:**
- Incident aggregate is larger — consider splitting if it becomes too large
- Investigation cannot be queried independently without the parent incident

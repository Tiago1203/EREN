# ADR-0201: Bounded Context Definitions

**Status:** Accepted

**Date:** 2026-07-16

**Deciders:** Architecture Board

---

## Context

Epic 2 implements 4 bounded contexts. We need to define the ownership, aggregates, and integration points for each.

---

## Decision

### Context 1: Device

**Owner:** Biomedical Domain
**Location:** `core/device/`

**Aggregate:** `Device`
- Identity: DeviceId, SerialNumber, ManufacturerInfo
- State: DeviceStatus, LocationInfo, CalibrationInfo, MaintenanceSchedule
- Lifecycle: registered → active → in_maintenance / calibration_due → out_of_service → decommissioned

**Invariants:**
- Serial number must be at least 3 characters
- Manufacturer name and model required
- Location (building, floor, room, department) required
- Calibration interval must be positive if set
- Maintenance type must be one of: preventive, corrective, predictive

**Domain Events:**
- `DeviceRegistered`
- `DeviceStatusChanged`
- `DeviceLocationChanged`

---

### Context 2: Incident

**Owner:** Biomedical Domain
**Location:** `core/incident/`

**Aggregate:** `EngineeringIncident`
- Identity: IncidentId, DeviceId, TenantId, EngineerId
- State: IncidentStatus, Priority, SafetyLevel, Symptom
- Lifecycle: REPORTED → TRIAGED → OPEN → IN_PROGRESS → RESOLVED → CLOSED

**Sub-aggregate:** `Investigation`
- Tracks: actions taken, evidence collected, notes
- Actions: investigation_started, measurement_taken, part_replaced, escalated, etc.

**Invariants:**
- Incident must have a device
- Incident must have a reporter
- Cannot close without at least one action
- Closed incidents cannot be modified
- Priority must be one of: critical, high, medium, low
- Safety classification: recommendation, monitoring, intervention, emergency

**Domain Events:**
- `IncidentReported`, `IncidentTriaged`, `IncidentOpened`
- `IncidentProgressed`, `IncidentEscalated`
- `IncidentResolved`, `IncidentClosed`

---

### Context 3: Recommendation

**Owner:** Biomedical Domain + AI Core (shared)
**Location:** `core/recommendation/`

**Aggregate:** `AIRecommendation`
- Identity: RecommendationId, IncidentId, DeviceId, TenantId
- State: RecommendationStatus, Confidence, Urgency, Category
- Lifecycle: GENERATED → PENDING_REVIEW → ACCEPTED / REJECTED / PARTIALLY_ACCEPTED / EXPIRED / SUPERSEDED

**Invariants:**
- Confidence score must be 0.0-1.0
- Cannot accept/reject expired recommendations
- Only pending recommendations can transition
- Expiry after 30 days by default

**Domain Events:**
- `RecommendationCreated`, `RecommendationAcceptedV2`, `RecommendationRejectedV2`
- `RecommendationFeedbackReceived`, `FeedbackReceived`

---

### Context 4: Knowledge

**Owner:** Clinical Domain
**Location:** `core/knowledge/`

**Aggregate:** `KnowledgeArticle`
- Identity: KnowledgeId, TenantId
- State: KnowledgeContent, Category, Tags
- Used for: evidence retrieval by AI

**Invariants:**
- Content must not be empty
- Category must be one of: device_manual, maintenance_procedure, safety_alert, regulatory, clinical_protocol
- Tags must be non-empty list

**Domain Events:** (none yet — articles are managed directly)

---

## Context Integration

```
Device Context ──publishes──▶ Incident Context
                                    │
                    ┌─────────────────┼─────────────────┐
                    ▼                 ▼                 ▼
              Knowledge Context  Recommendation Context  (WorkOrder — pending)
                    │                 │
                    └─────────subscribe to──────────────┘
```

**Integration:** Domain events via RabbitMQ. No synchronous calls between contexts.

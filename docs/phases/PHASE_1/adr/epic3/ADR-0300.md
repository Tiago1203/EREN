# ADR-0300: Hospital Context Architecture

**Status:** Accepted

**Date:** 2026-07-16

**Deciders:** Architecture Board

---

## Context

EREN serves hospital organizations with complex physical and organizational structures. The Hospital domain must model two independent hierarchies:

1. **Physical hierarchy:** Organization → Campus → Building → Floor → Room → Bed
2. **Organizational hierarchy:** Organization → Department → Unit

Epic 0 ADR-0050 established the conceptual model. This ADR defines the implementation architecture.

---

## Decision

**The Hospital domain is implemented as 6 bounded contexts:**

```
Hospital Ops team:
├── core/capacity/      — Bed, Room, Floor, Building, Campus
├── core/staffing/     — Staff, Role, Team, Shift, Schedule
├── core/organization/ — Hospital, Organization, multi-campus
├── core/department/   — Department, Unit
└── core/inventory/   — SparePart, Warehouse, PurchaseOrder, Supplier

Biomedical team:
└── core/asset/       — Asset, Manufacturer, Vendor, Contract, Warranty
                        (also: Maintenance plans — links to EPIC 2 Incident Context)
```

**Integration pattern:**
- Capacity Context → Device Context (device location via room_id)
- Capacity Context → Clinical Context (bed assignment via patient_id)
- Maintenance Context → Incident Context (linked work orders)
- Staffing Context → Capacity Context (staff assigned to unit)
- Staffing Context → Scheduling Context (scheduled shifts)
- Organization Context → Department Context (departments belong to organization)
- Asset Context → Device Context (asset tracks device history)
- Inventory Context → Asset Context (spare parts for assets)

---

## Physical Hierarchy

```
Organization
└── Campus (geographic cluster)
    └── Building
        └── Floor
            └── Room
                └── Bed (aggregate root — Capacity Context)
```

**All entities in the physical hierarchy:**
- `tenant_id` — multi-tenant isolation (all tables)
- `organization_id` — top-level organization
- `campus_id` — geographic cluster within organization
- `building_id` — building within campus
- `floor_id` — floor within building
- `room_id` — room within floor
- `bed_id` — bed within room

---

## Organizational Hierarchy

```
Organization
└── Department Group
    └── Department
        └── Unit
```

**All entities in the organizational hierarchy:**
- `tenant_id` — multi-tenant isolation (all tables)
- `organization_id` — top-level organization
- `department_group_id` — group of related departments
- `department_id` — department
- `unit_id` — unit within department

| Staffing | StaffRegistered, StaffAssigned, ShiftStarted, ShiftEnded, RoleAssigned |

| Inventory | SparePartAdded, SparePartConsumed, SparePartLowStock, PurchaseOrderCreated |

---

## Consequences

**Positive:**
- Clear ownership: Hospital Ops owns physical/org hierarchies, Biomedical owns assets
- Independent deployability per context
- Follows ADR-0050 conceptual model
- Parallel development of 6 contexts

**Negative:**
- Cross-context queries require integration events
- Physical hierarchy requires consistent ID propagation (tenant→org→campus→building→floor→room→bed)

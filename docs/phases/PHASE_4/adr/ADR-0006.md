# ADR-0006: EPIC 6 - Clinical RAG Architecture

## Status

**Accepted** - 2026-07-23

## Context

Necesitamos construir un pipeline RAG específico para clínico que:
1. Procese queries médicas de manera inteligente
2. Construya contexto clínico relevante
3. Empaquete evidencia con calidad
4. Genere prompts optimizados para LLMs

## Decision

### 1. Arquitectura de Query Processing

```
Query Module/
├── ClinicalQueryProcessor
├── QueryIntent
└── ProcessedQuery
```

**Responsabilidades:**
- Clasificación de tipo de query (diagnosis, treatment, etc.)
- Extracción de entidades médicas
- Expansion de términos
- Extracción de filtros

**Patrones de clasificación:**
- DIAGNOSIS: "diagnos", "condition", "symptoms"
- TREATMENT: "treatment", "therapy", "medication"
- DEVICE_USAGE: "device", "equipment", "operat"
- SAFETY_ALERT: "safety", "warning", "hazard"

### 2. Arquitectura de Context Building

```
Context Module/
├── ClinicalContextBuilder
├── ContextOptimizer
└── PromptContextGenerator
```

**Responsabilidades:**
- Filtrado por relevancia
- Límite de chunks y tokens
- Optimización para prompt
- Generación de texto formateado

**Configuración:**
```python
ContextConfig(
    max_chunks=5,
    max_tokens=4000,
    overlap_tokens=200,
    min_relevance=0.3,
)
```

### 3. Arquitectura de Evidence Building

```
Evidence Module/
├── ClinicalEvidenceBuilder
├── EvidenceItem
└── EvidencePackage
```

**Tipos de evidencia:**
```python
EvidenceType = {
    PEER_REVIEWED: "peer-reviewed publication",
    CLINICAL_GUIDELINE: "clinical guideline",
    REGULATORY: "regulatory document",
    MANUFACTURER: "manufacturer documentation",
    CASE_STUDY: "case study",
    EXPERT_OPINION: "expert opinion",
    WEB_CONTENT: "web content",
}
```

**Calidad:**
```python
EvidenceQuality = {
    HIGH: peer-reviewed, guideline, regulatory,
    MEDIUM: manufacturer docs, recent (< 5 years),
    LOW: web content, old sources,
}
```

### 4. Pipeline Integration

```
ClinicalRAGPipeline
├── Query Processor
├── Retrieval (EPIC 5)
├── Context Builder
├── Evidence Builder
└── Prompt Generator
```

## Consequences

### Positive

1. **Specialized**: Diseñado para clínica
2. **Quality-aware**: Evalúa calidad de evidencia
3. **Optimized**: Tokens limitados para prompts
4. **Composable**: Módulos independientes

### Negative

1. **Complexity**: Más componentes que RAG básico
2. **Latency**: Más pasos = más procesamiento

### Neutral

1. **Customization**: Templates de prompt configurables
2. **Extensibility**: Fácil añadir tipos de query

## Alternatives Considered

### 1. RAG genérico
- **Rejected**: No aprovecha conocimiento clínico

### 2. Sin Evidence Package
- **Rejected**: Sin trazabilidad de fuentes

### 3. Sin Query Processing
- **Rejected**: No se aprovecharía el contexto clínico

# ADR-0002: EPIC 2 - Knowledge Extraction Architecture

## Status

**Accepted** - 2026-07-23

## Context

Necesitamos convertir texto clínico no estructurado en conocimiento estructurado que pueda ser:
1. indexado en vector databases
2. razonado por el motor de inteligencia clínica
3. referenciado con citas verificables

El sistema debe:
1. Extraer entidades médicas (dispositivos, drogas, condiciones)
2. Extraer conceptos clínicos (diagnósticos, tratamientos)
3. Extraer relaciones entre entidades
4. Mapear terminología a ontologías estándar (SNOMED, ICD-10, MeSH)
5. Generar códigos médicos estandarizados

## Decision

### 1. Arquitectura de Extractors

Usamos extractores especializados por tipo de información:

```
Extractors/
├── EntityExtractor ───► BiomedicalEntity (DEVICE, DRUG, CONDITION, ...)
├── ConceptExtractor ──► BiomedicalConcept (DIAGNOSIS, TREATMENT, ...)
├── RelationExtractor ─► MedicalRelation (TREATS, CAUSES, ...)
└── MedicalNER ─────────► Combina todos para NER médico
```

**Ventajas:**
- Separación de concerns
- Testeable individualmente
- Reutilizable en diferentes pipelines

### 2. Arquitectura de Mappers

Mapeadores por ontología:

```
Mappers/
├── SNOMEDMapper ──► SNOMED CT (conceptos médicos)
├── UMLSMapper ─────► UMLS (lenguaje médico unificado)
├── MeSHMapper ─────► MeSH (encabezados de materia médica)
└── ICDMapper ───────► ICD-10 (códigos de enfermedades)
```

**Ventajas:**
- Cada mapper conoce las particularidades de su ontología
- Fácil de extender con nuevas ontologías
- TerminologyMapperFactory facilita uso

### 3. Arquitectura de Pipelines

Pipeline de extracción completo:

```python
class KnowledgeExtractionPipeline:
    async def extract(text, domain) -> ExtractedKnowledge:
        # 1. Extract entities
        entities = ner.recognize(text)
        
        # 2. Extract concepts
        _, concepts, relations = ner.extract_knowledge(text)
        
        # 3. Map to ontologies
        ontology_refs = await self._map_entities(entities)
        
        # 4. Extract medical codes
        codes = await self._extract_codes(ontology_refs)
        
        # 5. Calculate confidence
        confidence = self._calculate_confidence(entities, concepts)
        
        return ExtractedKnowledge(...)
```

### 4. Domain Objects

Definimos objetos especializados:

- `BiomedicalEntity` - Entidad con categoría, confianza, contexto
- `BiomedicalConcept` - Concepto clínico
- `MedicalRelation` - Relación entre entidades
- `OntologyReference` - Referencia a ontología
- `MappingResult` - Resultado de mapeo

### 5. Categorización Jerárquica

Usamos categorías jerárquicas:

```
EntityCategory/
├── DEVICE ────────── infusión pump, ventilator
├── DRUG ──────────── aspirin, metformin
├── CONDITION ──────── hypertension, diabetes
├── PROCEDURE ──────── surgery, biopsy
├── ANATOMY ───────── heart, lung
└── ...

ConceptCategory/
├── DIAGNOSIS ────────
├── TREATMENT ────────
├── SYMPTOM ──────────
└── ...

RelationCategory/
├── TREATS ───────────
├── CAUSES ───────────
├── INTERACTS_WITH ───
└── ...
```

## Consequences

### Positive

1. **Estructurado**: Conocimiento organizado en entidades, conceptos, relaciones
2. **Estandarizado**: Códigos médicos de ontologías reconocidas
3. **Rastreable**: Cada entidad tiene contexto y evidencia
4. **Extensible**: Fácil agregar nuevas categorías y ontologías
5. **Integrable**: Compatible con PHASE_2 y PHASE_3

### Negative

1. **Complejidad**: Múltiples componentes para mantener
2. **Dependencias**: Requiere bases de conocimiento de ontologías

### Neutral

1. **Rendimiento**: Procesamiento puede ser lento para grandes documentos
2. **Precisión**: Basado en regex y reglas, no ML

## Alternatives Considered

### 1. Un solo extractor genérico
- **Rejected**: No captura semántica de diferentes tipos de información

### 2. Solo regex sin ontologías
- **Rejected**: No proporciona códigos estandarizados

### 3. NER basado en ML (spaCy, HuggingFace)
- **Rejected**: Requiere entrenamiento, menos control
- Considerar para futura versión con fine-tuning

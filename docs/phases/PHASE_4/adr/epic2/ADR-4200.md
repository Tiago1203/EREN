# ADR-4200: Qdrant Client Architecture

## Estado
**Proposed** 📝

## Contexto
Necesitamos almacenar y buscar vectores de conocimiento clínico en Qdrant. El cliente debe:
- Soportar múltiples colecciones especializadas
- Integrarse con PHASE_2 cognitive/rag
- Proveer búsqueda híbrida (dense + sparse)

## Decisión

### Arquitectura de Colecciones

| Colección | Descripción | Vector Size |
|-----------|-------------|-------------|
| medical_literature | PubMed, estudios clínicos | 768 |
| device_manuals | Manuales de equipos | 768 |
| clinical_guidelines | Guías clínicas | 768 |
| regulatory_docs | Documentos regulatorios | 768 |
| knowledge_articles | Artículos KB | 768 |
| evidence_base | Base de evidencia | 768 |

### Búsqueda Híbrida

```
Query Text
    │
    ├──→ Dense Search (embeddings) ──→ Vector Store
    │
    └──→ Sparse Search (BM25) ──────→ Keyword Index
                │
                └──→ RRF Fusion ───→ Results
```

### Configuración de Distancia

Usar **Cosine** como métrica de distancia por defecto para embeddings normalizados.

## Consecuencias

### Positivas
- Colecciones especializadas para cada tipo de conocimiento
- Búsqueda híbrida combina strengths de dense y sparse
- Compatible con PHASE_2 cognitive/rag

### Negativas
- Requiere Qdrant como dependencia
- Multiple collections requieren más gestión

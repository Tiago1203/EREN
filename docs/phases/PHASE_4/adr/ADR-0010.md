# ADR-0010: EPIC 10 - Knowledge Synchronization Architecture

## Status

**Accepted** - 2026-07-23

## Context

Necesitamos mantener actualizado el conocimiento biomédico:
1. Sincronizar fuentes externas
2. Programar actualizaciones
3. Monitorear salud de fuentes
4. Detectar cambios automáticamente

## Decision

### 1. Arquitectura de Sync

```
Sync/
├── SyncJob ──────────────────────► Job model
├── InMemorySyncManager ───────────► Manager
├── UpdatePackage ─────────────────► Update package
└── DeltaProcessor ───────────────► Delta processing
```

**Fuentes:**
- PUBMED, FDA, EMA, CLINICAL_GUIDELINES, DEVICE_MANUFACTURERS

### 2. Arquitectura de Schedule

```
Schedule/
├── ScheduleConfig ────────────────► Config
├── InMemoryScheduler ────────────► Scheduler
└── CronParser ───────────────────► Cron parsing
```

**Frecuencias:**
- HOURLY, DAILY, WEEKLY, MONTHLY, ON_DEMAND

### 3. Arquitectura de Monitor

```
Monitor/
├── SourceHealth ─────────────────► Health status
├── InMemorySourceMonitor ─────────► Monitor
├── HealthChecker ─────────────────► Health check
└── ChangeDetector ───────────────► Change detection
```

**Estados:**
- HEALTHY, DEGRADED, DOWN, UNKNOWN

## Consequences

### Positive

1. **Automation**: Sync automático
2. **Scheduling**: Control de timing
3. **Monitoring**: Salud de fuentes
4. **Reliability**: Deltas pequeños

### Negative

1. **Complexity**: Más componentes
2. **Dependencies**: Depende de fuentes externas

### Neutral

1. **In-memory**: Para demo; production requiere DB
2. **Polling**: Basado en polling (no webhooks)

## Alternatives Considered

### 1. Sin sync automático
- **Rejected**: Conocimiento se vuelve obsoleto

### 2. Webhooks en vez de polling
- **Rejected**: No todas las fuentes suportan

### 3. Sync bajo demanda solo
- **Rejected**: No hay actualizaciones proactivas

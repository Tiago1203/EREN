# ADR-0004: EPIC 4 - Vector Indexing Architecture

## Status

**Accepted** - 2026-07-23

## Context

Necesitamos indexar conocimiento biomédico en un vector database (Qdrant) para permitir:
1. Búsqueda semántica de vectores
2. Filtrado por payload
3. Retrieval de alta velocidad
4. Escalabilidad horizontal

## Decision

### 1. Arquitectura de Qdrant

Usamos wrapper sobre cliente Qdrant con fallback:

```
QdrantClientWrapper
├── QdrantClient (real) ──► Producción
└── InMemoryQdrantClient ──► Testing, desarrollo
```

**Ventajas:**
- Abstraction del cliente real
- Fácil testing
- Configuración centralizada

### 2. Gestión de Colecciones

Colecciones por tipo de contenido:

```
Collections/
├── knowledge ──────► Documentos médicos
├── entities ───────► Entidades biomédicas
├── concepts ────────► Conceptos clínicos
└── embeddings ─────► Embeddings simples
```

**Ventajas:**
- Aislamiento de datos
- Optimización por tipo
- Queries específicas

### 3. Payload Estructurado

Payloads con schema definido:

```python
VectorPayload
├── Identificación (id, source_id)
├── Contenido (text, title)
├── Metadatos (domain, source_type)
├── Códigos médicos (icd, snomed, mesh)
└── Calidad (quality_score, confidence_score)
```

**Ventajas:**
- Filtrado eficiente
- Metadata rica
- Trazabilidad

### 4. Builders de Payload

Builders especializados:

```
PayloadBuilders/
├── KnowledgePayloadBuilder ──► Documentos
└── EntityPayloadBuilder ──────► Entidades
```

### 5. Templates de Colección

Pre-configuraciones optimizadas:

```python
CollectionFactory.TEMPLATES = {
    "knowledge": CollectionConfig(m=16, ef_construct=200),
    "entities": CollectionConfig(m=12, ef_construct=150),
    "concepts": CollectionConfig(m=16, ef_construct=200),
}
```

## Consequences

### Positive

1. **Rendimiento**: HNSW con parámetros optimizados
2. **Escalabilidad**: Sharding y replicación
3. **Testing**: InMemoryQdrantClient para tests
4. **Flexibilidad**: Múltiples tipos de colección

### Negative

1. **Complejidad**: Más componentes
2. **Ops**: Requiere Qdrant desplegado

### Neutral

1. **Costo**: Qdrant cloud o self-hosted
2. **Latencia**: Depende de la red

## Alternatives Considered

### 1. Solo un cliente real
- **Rejected**: Difícil testing

### 2. Sin payload estructurado
- **Rejected**: Sin metadata, sin filtrado

### 3. Colección única
- **Rejected**: Mezcla de tipos, difícil optimizar

## Implementation Notes

### QdrantClientWrapper

```python
class QdrantClientWrapper(BaseQdrantClient):
    def __init__(self, config: QdrantConfig):
        self.config = config
    
    def _get_client(self):
        try:
            from qdrant_client import QdrantClient
            return QdrantClient(host=..., port=...)
        except ImportError:
            return InMemoryQdrantClient()
```

### PayloadBuilder

```python
class KnowledgePayloadBuilder(BasePayloadBuilder):
    def build(self, data: dict) -> VectorPayload:
        return VectorPayload(
            text=data["text"],
            source_id=data["source_id"],
            domain=data.get("domain", ""),
            ...
        )
```

# ADR-4300: Knowledge Retriever Design

## Estado
**Proposed** 📝

## Contexto
El recuperador de conocimiento debe:
- Buscar en múltiples fuentes (PHASE_1, PHASE_3, Qdrant)
- Aplicar filtros clínicos (especialidad, codes, dispositivos)
- Retornar resultados con scoring de relevancia
- Conectar con Evidence Retrieval de PHASE_3

## Decisión

### Fuentes de Conocimiento

| Fuente | Tipo | Integración |
|--------|------|-------------|
| Knowledge Articles | PHASE_1 | Repository |
| Device Manuals | Qdrant | Vector Store |
| Clinical Guidelines | Qdrant | Vector Store |
| Evidence Base | PHASE_3 | Evidence Store |
| Standards | Qdrant | Vector Store |

### Pipeline de Retrieval

```
Query
    │
    ├──→ Vector Search (Qdrant) ──→ Chunk Results
    │
    ├──→ Evidence Search (PHASE_3) ─→ Evidence Results
    │
    └──→ Knowledge Search (PHASE_1) ─→ Article Results
                │
                └──→ Fusion & Ranking ──→ Final Results
```

### Filtros Clínicos

- Medical specialty
- Device categories
- ICD/SNOMED codes
- Evidence levels
- Date ranges

## Consecuencias

### Positivas
- Busca en todas las fuentes relevantes
- Filtros específicos para dominio clínico
- Integración nativa con PHASE_1 y PHASE_3

### Negativas
- Más compleja la orquestación de fuentes
- Potencial latencia por múltiples queries

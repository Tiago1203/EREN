# ADR-0001: EPIC 1 - Document Processing Architecture

## Status

**Accepted** - 2026-07-23

## Context

Necesitamos procesar documentos biomédicos de múltiples formatos (PDF, DOCX, HTML, Markdown, TXT) antes de convertirlos en conocimiento estructurado.

El procesamiento debe:
1. Extraer texto de múltiples formatos
2. Detectar y estructurar tablas
3. Extraer metadata médica especializada (DOI, PMID, ICD codes)
4. Normalizar terminología médica
5. Detectar idioma

## Decision

### 1. Arquitectura de Parsers

Usamos una arquitectura de **Parser Factory** con parsers especializados:

```
ParserFactory
├── PDFParser (PyPDF2, pdfplumber)
├── DOCXParser (python-docx)
├── HTMLParser (BeautifulSoup)
├── MarkdownParser (Custom)
└── TextParser (Built-in)
```

**Ventajas:**
- Cada parser es especializado para su formato
- Fácil de extender con nuevos formatos
- ParserFactory selecciona el parser correcto

### 2. Arquitectura de Extractors

Extractors independientes que procesan el contenido ya parseado:

```
Content
├── TableExtractor ───→ Tables
├── FigureExtractor ──→ Figures
├── MetadataExtractor ─→ MedicalMetadata
├── SectionExtractor ─→ Sections
└── LanguageDetector ─→ Language
```

### 3. Arquitectura de Normalizers

Normalizers compuestos que se pueden encadenar:

```python
CompositeNormalizer
├── MedicalCleaner (símbolos médicos)
├── ContentCleaner (headers/footers)
├── StructureNormalizer (headers markdown)
└── TerminologyNormalizer (abreviaturas)
```

### 4. Domain Objects

Definimos objetos especializados para el dominio biomédico:

- `BiomedicalDocument` - Documento normalizado
- `MedicalMetadata` - Metadata médica (DOI, PMID, ICD, SNOMED)
- `Table` - Tabla estructurada
- `Figure` - Figura con caption
- `Section` - Sección del documento

### 5. Integración con EPIC 0

Usamos Foundation de EPIC 0:
- `DocumentFormat` enum
- `ProcessingStatus` enum
- `DocumentParseError`, `UnsupportedFormatError` excepciones
- `EventType` para eventos de dominio
- `InMemoryEventPublisher` para pub/sub

## Consequences

### Positive

1. **Modularidad**: Cada componente tiene responsabilidad única
2. **Extensibilidad**: Fácil agregar nuevos parsers/normalizers
3. **Testabilidad**: Cada componente es unit-testable
4. **Reutilización**: Extractors reutilizables
5. **Rendimiento**: Procesamiento en pipeline paralelo

### Negative

1. **Complejidad**: Más componentes que una solución monolítica
2. **Overhead**: Más abstracciones para casos simples

## Alternatives Considered

### 1. Parser único con condicionales
- **Rejected**: Viola SRP, difícil de mantener

### 2. Librería de terceros (pdfminer, textract)
- **Rejected**: Menos control, dependencias externas

### 3. Solo un normalizer genérico
- **Rejected**: No captura domain knowledge médico

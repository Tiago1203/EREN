# ADR-0000: EPIC 0 - Knowledge Infrastructure Foundation Architecture

## Status

**Accepted** - 2026-07-23
**Updated** - 2026-07-23 (v2 - Post-Audit Fixes)

## Context

Necesitamos establecer una infraestructura base (Shared Kernel) que soporte todos los EPICs de PHASE 4 y proporcione integración limpia con PHASE 1, 2, y 3.

El objetivo es:
1. Evitar duplicación de código entre EPICs
2. Proveer tipos y contratos comunes
3. Centralizar configuración
4. Facilitar testing
5. Permitir evolución independiente de EPICs

### Post-Audit Changes (v2)

**Problemas corregidos:**
- CRÍTICO-001: EvidenceLevel ahora importado desde PHASE_3 (sistema Oxford unificado)
- CRÍTICO-002: PHASE1Gateway, PHASE2Gateway, PHASE3Gateway implementados con conexión real
- CRÍTICO-003: PHASE_4 ahora depende de PHASE_3 mediante gateways
- IMP-001: QualityDimension importado desde PHASE_3

## Decision

### 1. Estructura de Módulos

```
core/PHASE_4/foundation/
├── __init__.py              # Exports principales
├── constants/               # Constantes globales
├── config/                  # Configuración
├── exceptions/              # Excepciones
└── events/                 # Eventos de dominio
```

### 2. Tipos Comunes

Definimos enums, value objects, domain objects, y entities en el módulo principal para evitar imports circulares.

**Enums**:
- `DocumentFormat` - Formatos de documento
- `KnowledgeDomain` - Dominios clínicos
- `ProcessingStatus` - Estados de procesamiento
- `QualityLevel` - Niveles de calidad
- `EvidenceLevel` - Niveles de evidencia médica (Oxford)
- `GovernanceStatus` - Estados de gobernanza

**Value Objects** (inmutables):
- `EmbeddingVector`
- `SourceReference`
- `Citation`
- `MedicalCode`

**Domain Objects**:
- `KnowledgeSource`
- `KnowledgeMetadata`
- `KnowledgeChunk`
- `KnowledgeCollection`
- `RetrievalContext`
- `SearchRequest`

**Entities**:
- `ProcessedDocument`

Proveemos implementaciones base que pueden ser extendidas:
- `BaseKnowledgeService`
- `BaseRetriever`
- `BaseIndexer`
- `BaseChunker`
- `BaseEmbeddingProvider`

### 5. Gateways

Creamos gateways explícitos para integración con otras fases:
- `PHASE1Gateway` - Consume Device, Knowledge, Incident contexts
- `PHASE2Gateway` - Extiende embeddings, retrieval
- `PHASE3Gateway` - Integra reasoning, evidence
- `PHASE5Contract` - Output hacia PHASE 5

## Consequences

### Positive

1. **Consistencia**: Todos los EPICs usan los mismos tipos y contratos
2. **Reutilización**: Base implementations reducen código duplicado
3. **Testing**: Contracts facilitan mocking
4. **Evolución**: EPICs pueden evolucionar independientemente
5. **Documentación**: Estructura clara facilita onboarding

### Negative

1. **Acoplamiento inicial**: Todos los EPICs dependen del foundation
2. **Cambio de scope**: Cambios en foundation afectan a todos

### Neutral

1. Imports más largos pero más explícitos
2. Más archivos pero mejor organizados

## Alternatives Considered

### 1. Módulos separados por EPIC
- **Rejected**: Causaría duplicación de tipos comunes

### 2. Un solo archivo con todo
- **Rejected**: Difícil de mantener y navegar

### 3. Paquetes Python separados
- **Rejected**: Over-engineering para el scope actual

## Notes

- El foundation es el único módulo que todos los demás EPICs deben importar
- Los gateways son lazy-loaded para evitar ciclos de importación
- Las implementaciones base son opcionales pero recomendadas
- Los eventos usan Pub/Sub para desacoplar componentes

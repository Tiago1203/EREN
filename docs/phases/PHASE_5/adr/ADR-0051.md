# ADR-0051: Multi-Agent System Architecture

*Status: Proposed*
*Date: 2026-07-23*
*Deciders: Architecture Board*

---

## Context

EREN PHASE 4 ha completado la plataforma de conocimiento biomédico. PHASE 5 debe transformar esta plataforma en un sistema cognitivo distribuido donde múltiples agentes especializados colaboran para resolver problemas complejos de Ingeniería Clínica.

El sistema debe:
- Soportar múltiples agentes especializados
- Permitir comunicación entre agentes
- Gestionar ciclo de vida de agentes
- Mantener contexto compartido
- Orquestar ejecución de tareas
- Garantizar explicabilidad y auditoría

---

## Decision

Se adopta una **arquitectura de agentes con shared kernel** donde:

### 1. Shared Kernel (EPIC 0)

Todos los EPICs comparten:
- **Agent Base Types**: Agent, AgentTask, AgentMessage, AgentContext
- **Agent Contracts**: IAgent, IAgentRegistry, IMessageBroker, ILifecycleManager
- **Agent Events**: Sistema de eventos centralizado
- **Agent Gateway**: Acceso a FASE 1-4

### 2. Agente como Unidad Básica

Cada agente es:
- **Autónomo**: Puede ejecutarse independientemente
- **Especializado**: Tiene capabilities específicas
- **Stateful**: Mantiene estado interno
- **Communicative**: Se comunica via mensajes
- **Auditable**: Todas las acciones son registradas

### 3. Orquestador Central (EPIC 1)

El orquestador:
- Selecciona agentes según capabilities
- Planifica ejecución de tareas
- Gestiona dependencias
- Sintetiza respuestas
- Evalúa confianza

### 4. Message Broker

Comunicación asíncrona via broker:
- **Topcis**: Canales de comunicación
- **Publish/Subscribe**: Modelo de mensajería
- **Guaranteed Delivery**: Mensajes garantizados
- **Dead Letter Queue**: Manejo de errores

### 5. Ciclo de Vida

Estados de agente:
```
INITIAL → IDLE → RUNNING → WAITING → COMPLETED
                  ↓
                FAILED → STOPPED
```

---

## Consequences

### Positive

- **Modularidad**: Cada agente es independiente
- **Escalabilidad**: Fácil agregar nuevos agentes
- **Testabilidad**: Agentes testeables aisladamente
- **Reusabilidad**: Capabilities compartidas
- **Explicabilidad**: Todas las acciones son trazables

### Negative

- **Complejidad**: Sistema distribuido es complejo
- **Latencia**: Comunicación entre agentes añade latencia
- **Consistencia**: Consistencia eventual vs. fuerte
- **Debugging**: Difícil debugging distribuido

### Neutral

- **Estado**: Necesita gestión de estado distribuido
- **Monitoreo**: Requiere observabilidad completa
- **Documentación**: Más documentación necesaria

---

## Alternatives Considered

### Alternative 1: Monolito con Módulos de Agente

```
Agente1 │ Agente2 │ Agente3 │ ... │ AgenteN
              │
         Shared Kernel
```

**Pros**: Más simple, menos latencia
**Cons**: Menos escalable, acoplamiento

### Alternative 2: Microservicios Independientes

```
Agente1    Agente2    Agente3
  │          │          │
ServiceBus ServiceBus ServiceBus
```

**Pros**: Máxima independencia, escalabilidad
**Cons**: Complejidad operacional, debugging difícil

### Alternative 3: Actor Model (Akka/Erlang)

**Pros**: Modelo de concurrencia probado
**Cons**: Paradigma diferente, curva de aprendizaje

# ADR-0064: PHASE 5 Refactoring - Architecture Improvements

*Status: ACCEPTED*
*Date: 2026-07-23*
*Deciders: Architecture Board*

---

## Context

Después de la auditoría de PHASE 5, se identificaron los siguientes problemas críticos:

1. **Duplicación de Domain Objects** - 6 objetos duplicados entre EPICs
2. **Sin comunicación entre agentes** - AgentBus no implementado
3. **Gateways vacíos** - Sin integración real con FASE 1-4
4. **Sin concatenación de EPICs** - Orchestrator no conectaba EPICs
5. **Sin registry de agentes** - Agentes no descubribles

---

## Decision

Se implementaron las siguientes mejoras:

### 1. Shared Domain Objects

Centralización de Domain Objects duplicados en `core.PHASE_5.foundation.domain.shared`:

```python
# Shared Objects
SharedSessionStatus    # (era duplicado en EPIC 7, 10)
SharedMetricType       # (era duplicado en EPIC 1, 10)
SharedAgentMetric      # (era duplicado en EPIC 1, 10)
SharedRecommendation    # (era duplicado en EPIC 1, 10)
SharedLearningSession   # (era duplicado en EPIC 1, 10)
SharedOptimizationReport # (era duplicado en EPIC 1, 10)
```

### 2. AgentBus

Implementación de sistema de mensajería:

```python
class AgentBus:
    async def send(sender, receiver, action, payload)
    async def broadcast(sender, action, payload)
    async def subscribe(agent_id, topic, handler)
    async def publish(publisher, topic, action, payload)
    async def receive(agent_id)
```

### 3. IntegratedPhaseGateway

Gateways con integración real:

```python
class IntegratedMultiPhaseGateway:
    async def initialize_all()
    async def get_full_context(query, context_type)
    
    # Acceso a todas las fases
    phase1: IntegratedPhase1Gateway  # Business Domain
    phase2: IntegratedPhase2Gateway  # Cognitive OS
    phase3: IntegratedPhase3Gateway  # Clinical Intelligence
    phase4: IntegratedPhase4Gateway  # Knowledge Platform
```

### 4. OrchestratorAgent

Agente orquestador que conecta EPICs:

```python
class OrchestratorAgent(BaseAgent):
    # Registro de EPICs
    _epic_registry: EpicRegistry
    
    # Conexión con otros EPICs
    _bus: AgentBus
    
    # Flujo determinado por tipo de tarea
    _determine_epic_flow(task) -> list[str]
    
    # Ejecución orquestada
    async def execute(task) -> AgentResult
```

### 5. Flujo de EPICs

El OrchestratorAgent determina el flujo basado en el tipo de tarea:

| Tipo | EPICs |
|------|-------|
| biomedical | 1 → 2 → 11 |
| diagnostic | 1 → 2 → 3 → 11 |
| knowledge | 1 → 4 → 11 |
| research | 1 → 4 → 5 → 11 |
| planning | 1 → 6 → 11 |
| collaboration | 1 → 7 → 8 → 11 |
| memory | 1 → 9 → 10 → 11 |
| governance | 1 → 11 |
| default | 1 → 2 → 3 → ... → 11 |

---

## Consequences

### Positive

1. **Eliminación de duplicación DRY** - 6 objetos centralizados
2. **Comunicación entre agentes** - AgentBus permite mensajería
3. **Integración real con FASE 1-4** - Gateways funcionales
4. **Concatenación de EPICs** - Orchestrator conecta flujo
5. **Mejora en tests** - 370 tests passing

### Negative

1. **Mayor complejidad inicial** - Más componentes
2. **Curva de aprendizaje** - Más conceptos

### Neutral

1. **Extensibilidad** - Facilita agregar nuevos EPICs
2. **Mantenibilidad** - Código más limpio

# ADR-0055: Diagnostic Agent Architecture

*Status: Proposed*
*Date: 2026-07-23*
*Deciders: Architecture Board*

---

## Context

EPIC 1 (Agent Orchestrator) necesita agentes especializados para diagnosticar fallas técnicas. EPIC 2 (Biomedical Agent) proporciona contexto de equipos médicos. EPIC 3 debe crear el **Diagnostic Agent** - un agente especializado en:

- Analizar fallas de equipos médicos
- Generar hipótesis diagnósticas
- Identificar causa raíz (RCA)
- Construir planes de diagnóstico sistemático
- Correlacionar eventos de falla

El diseño debe permitir que el DiagnosticAgent:
- Use técnicas de RCA (5 Whys, Fishbone, Fault Tree)
- Genere planes de diagnóstico paso a paso
- Correlacione eventos para identificar causa común
- Se integre con EPIC 2 para contexto de equipos

---

## Decision

Se adopta una **arquitectura de agente con motores especializados**:

### 1. DiagnosticAgent como Fachada

```
┌────────────────────────────────────────────────────────────────┐
│                      DiagnosticAgent                             │
│  (Fachada - recibe tareas, delega a motores)                   │
└──────────────────────────┬─────────────────────────────────────┘
                           │
     ┌─────────────────────┼─────────────────────┐
     │                     │                     │
     ▼                     ▼                     ▼
┌──────────────┐    ┌───────────────┐    ┌──────────────────┐
│   Failure    │    │   RootCause  │    │   Diagnostic    │
│  Analyzer    │    │   Analyzer   │    │    Planner      │
└──────────────┘    └───────────────┘    └──────────────────┘
```

**Ventajas:**
- Agente único visible hacia el Orchestrator
- Motores son detalles de implementación
- Fácil agregar/quitar motores
- Testing simplificado

### 2. Motores Especializados

| Motor | Responsabilidad |
|-------|-----------------|
| `FailureAnalyzer` | Análisis de modos de falla |
| `RootCauseAnalyzer` | Análisis de causa raíz |
| `DiagnosticPlanner` | Planificación de diagnóstico |
| `FaultCorrelator` | Correlación de fallas |

### 3. Task Types

```python
class DiagnosticTaskType(Enum):
    FAILURE_ANALYSIS = "failure_analysis"
    ROOT_CAUSE = "root_cause"
    PREDICTIVE = "predictive"
    CORRELATION = "correlation"
    TROUBLESHOOTING = "troubleshooting"
    INVESTIGATION = "investigation"
```

### 4. Domain Objects

```python
# Tarea de diagnóstico
class DiagnosticTask:
    task_id: str
    task_type: DiagnosticTaskType
    device_id: str
    symptom: str
    error_codes: list[str]

# Patrón de falla
class FailurePattern:
    pattern_id: str
    severity: FailureSeverity
    known_causes: list[str]

# Reporte de diagnóstico
class DiagnosticReport:
    report_id: str
    diagnosis: str
    confidence: DiagnosisConfidence
    primary_cause: str
```

---

## Consequences

### Positive

1. **Especialización**: Agente enfocado en diagnóstico
2. **Modularidad**: Motores son independientes
3. **Testabilidad**: Cada motor testeable aisladamente
4. **Reusabilidad**: Motores pueden ser reutilizados
5. **Extensibilidad**: Fácil agregar nuevos tipos de análisis

### Negative

1. **Acoplamiento**: Motores acoplados al agente
2. **Complejidad**: Más clases que un agente simple

### Neutral

1. **Estado**: Motores son stateless por diseño
2. **Configuración**: Cada motor configurable

---

## Alternatives Considered

### Alternative 1: Agente Monolítico

```python
class DiagnosticAgent:
    async def analyze_failure(self, ...): ...
    async def find_root_cause(self, ...): ...
    async def correlate_faults(self, ...): ...
```

**Pros**: Simple, sin delegation
**Cons**: Viola SRP, difícil de mantener

### Alternative 2: Multi-Agent Diagnosis

```
DiagnosticAgent → FailureAgent + RCAAgent + CorrelationAgent
```

**Pros**: Máximo desacoplamiento
**Cons**: Overhead de mensajería, más complejidad para Orchestrator

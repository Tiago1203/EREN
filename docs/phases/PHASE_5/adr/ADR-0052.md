# ADR-0052: Agent Communication Protocol

*Status: Proposed*
*Date: 2026-07-23*
*Deciders: Architecture Board*

---

## Context

El sistema multiagente de PHASE 5 requiere un protocolo de comunicación robusto que permita:

- **Comunicación síncrona**: Request/Response para queries directas
- **Comunicación asíncrona**: Publish/Subscribe para notificaciones
- **Broadcasting**: Difusión de eventos a múltiples agentes
- **Guaranteed Delivery**: Mensajes no se pierden
- **Dead Letter Handling**: Manejo de mensajes fallidos
- **Message Ordering**: Orden de mensajes preservado

---

## Decision

Se adopta un **protocolo de mensajería híbrio** combinando:

### 1. Message Types

```python
class MessageType(Enum):
    # Síncrono
    REQUEST = "request"        # Request con Response esperada
    RESPONSE = "response"     # Response a Request
    
    # Asíncrono
    NOTIFICATION = "notification"  # Pub/Sub notificación
    EVENT = "event"            # Evento del sistema
    
    # Control
    ERROR = "error"           # Mensaje de error
    HEARTBEAT = "heartbeat"   # Heartbeat alive
    COMMAND = "command"        # Comando directo
```

### 2. Message Structure

```python
@dataclass
class AgentMessage:
    # Identificación
    message_id: str           # UUID único
    correlation_id: str       # Para tracking de conversación
    parent_id: str | None    # Mensaje padre (reply-to)
    
    # Routing
    sender: str               # Agent ID emisor
    receiver: str             # Agent ID receptor (o wildcard)
    topic: str | None         # Topic para Pub/Sub
    
    # Contenido
    type: MessageType
    action: str               # Acción específica
    payload: dict             # Datos del mensaje
    metadata: dict            # Metadatos adicionales
    
    # Quality of Service
    priority: AgentPriority
    ttl: int                  # Time-to-live en segundos
    retry_count: int = 0
    max_retries: int = 3
    
    # Timestamps
    created_at: datetime
    sent_at: datetime | None
    received_at: datetime | None
    processed_at: datetime | None
```

### 3. Communication Patterns

#### Pattern 1: Request/Response (Síncrono)

```python
# Agente A solicita a Agente B
message = AgentMessage(
    sender="agent_a",
    receiver="agent_b",
    type=MessageType.REQUEST,
    action="diagnose",
    payload={"device_id": "dev_123"},
)
response = await broker.send_and_wait(message, timeout=30)
```

#### Pattern 2: Publish/Subscribe (Asíncrono)

```python
# Agente B se subscribe a tópico
await broker.subscribe("agent_b", topics=["device_events", "alerts"])

# Agente A publica en tópico
message = AgentMessage(
    sender="agent_a",
    topic="device_events",
    type=MessageType.NOTIFICATION,
    action="device_status_changed",
    payload={"device_id": "dev_123", "status": "offline"},
)
await broker.publish(message)
```

#### Pattern 3: Broadcasting

```python
# Difundir a todos los agentes
message = AgentMessage(
    sender="orchestrator",
    receiver="*",  # Wildcard = todos
    type=MessageType.EVENT,
    action="system_shutdown",
    payload={"reason": "maintenance"},
)
await broker.broadcast(message)
```

#### Pattern 4: Fan-Out

```python
# Enviar a múltiples agentes
message = AgentMessage(
    sender="orchestrator",
    receivers=["biomedical", "diagnostic", "knowledge"],  # Lista específica
    type=MessageType.REQUEST,
    action="analyze",
    payload={"query": "..."},
)
results = await broker.send_to_many(message)
```

### 4. Quality of Service (QoS)

| Level | Guarantee | Use Case |
|-------|----------|----------|
| QoS 0 | At most once | Heartbeats, non-critical |
| QoS 1 | At least once | Notifications |
| QoS 2 | Exactly once | Critical commands |

### 5. Error Handling

```python
class MessageError(Enum):
    TIMEOUT = "timeout"              # No response within timeout
    AGENT_NOT_FOUND = "agent_not_found"  # Receiver doesn't exist
    QUEUE_FULL = "queue_full"        # Receiver queue full
    INVALID_MESSAGE = "invalid_message"  # Malformed message
    PROCESSING_FAILED = "processing_failed"  # Processing error
```

Dead Letter Queue (DLQ) para mensajes fallidos:
- Mensajes que exceden `max_retries` van a DLQ
- DLQ es auditable y recuperable
- Admin puede re-procesar desde DLQ

### 6. Message Broker Implementation

```python
class MessageBroker:
    """Broker de mensajes para sistema multiagente."""
    
    # Cola por agente
    _queues: dict[str, deque[AgentMessage]]
    
    # Topics para Pub/Sub
    _topics: dict[str, set[str]]  # topic -> set of agent_ids
    
    # Dead Letter Queue
    _dlq: deque[AgentMessage]
    
    async def send(self, message: AgentMessage) -> bool:
        """Envía mensaje síncrono."""
    
    async def publish(self, message: AgentMessage) -> bool:
        """Publica a todos los subscribers del topic."""
    
    async def send_and_wait(
        self,
        message: AgentMessage,
        timeout: int = 30
    ) -> AgentMessage:
        """Envía y espera respuesta."""
    
    async def subscribe(
        self,
        agent_id: str,
        topics: list[str]
    ) -> bool:
        """Suscribe agente a topics."""
    
    async def get_queue(self, agent_id: str) -> list[AgentMessage]:
        """Obtiene mensajes pendientes para agente."""
```

## Consequences

### Positive

- **Flexibilidad**: Múltiples patrones de comunicación
- **Escalabilidad**: Broker centralizado permite scaling
- **Debugging**: Mensajes son trazables
- **Guaranteed Delivery**: QoS configurable
- **Resilience**: DLQ para mensajes fallidos

### Negative

- **Latencia**: Broker añade latencia
- **Complexity**: Más componentes
- **Single Point**: Broker es punto único de falla

### Neutral

- **Storage**: Mensajes deben persistirse para auditing
- **Monitoring**: Broker necesita métricas

---

## Alternatives Considered

### Alternative 1: Direct Communication (Point-to-Point)

```
Agent A ───────► Agent B
```

**Pros**: Menor latencia, simpler
**Cons**: N agent connections = N*(N-1)/2, tightly coupled

### Alternative 2: gRPC Streaming

**Pros**: Protocol buffer, streaming nativo
**Cons**: Menos flexible para Pub/Sub

### Alternative 3: Actor Model (Akka)

**Pros**: Built-in messaging
**Cons**: Vendor lock-in

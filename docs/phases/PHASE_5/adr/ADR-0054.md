# ADR-0054: Biomedical Agent Architecture

*Status: Proposed*
*Date: 2026-07-23*
*Deciders: Architecture Board*

---

## Context

EPIC 1 (Agent Orchestrator) necesita invocar agentes especializados para resolver solicitudes clínicas. EPIC 2 debe crear el **Biomedical Agent** - un agente especializado en Ingeniería Clínica que pueda:

- Analizar equipos médicos
- Generar recomendaciones de mantenimiento (preventivo, correctivo, predictivo)
- Verificar cumplimiento de normas (IEC 60601-1, ISO 14971, etc.)
- Proporcionar información de fabricantes
- Gestionar calibraciones
- Evaluar riesgos de dispositivos

El diseño debe permitir que el BiomedicalAgent sea:
- **Auto-contenido**: No necesita otros agentes para completar sus tareas
- **Especializado**: Solo maneja Ingeniería Clínica
- **Composable**: Fácil de extender con nuevos expertos
- **Integrable**: Compatible con EPIC 1 (Orchestrator)

---

## Decision

Se adopta una **arquitectura de agente con expertos internos**:

### 1. BiomedicalAgent como Fachada

```
┌────────────────────────────────────────────────────────────────┐
│                      BiomedicalAgent                            │
│  (Fachada - recibe tareas, delega a expertos)                  │
└──────────────────────────┬─────────────────────────────────────┘
                           │
     ┌─────────────────────┼─────────────────────┐
     │                     │                     │
     ▼                     ▼                     ▼
┌─────────┐        ┌─────────────┐        ┌────────────┐
│Equipment│        │ Maintenance │        │ Standards  │
│ Expert  │        │   Expert   │        │   Expert   │
└─────────┘        └─────────────┘        └────────────┘
```

**Ventajas:**
- Agente único visible hacia el Orchestrator
- Expertos son detalles de implementación
- Fácil agregar/quitar expertos
- Testing simplificado

### 2. Expertos como Servicios Internos

Cada experto es una clase con responsabilidad única:

| Experto | Responsabilidad |
|---------|-----------------|
| `EquipmentExpert` | Análisis técnico de equipos |
| `MaintenanceExpert` | Recomendaciones de mantenimiento |
| `StandardsExpert` | Normas IEC/ISO/AAMI |
| `ManufacturerExpert` | Información de fabricantes |
| `CalibrationExpert` | Calibraciones |

### 3. Task Types

El agente procesa diferentes tipos de tareas:

```python
class BiomedicalTaskType(Enum):
    DEVICE_ANALYSIS = "device_analysis"
    MAINTENANCE_PLANNING = "maintenance_planning"
    CALIBRATION_CHECK = "calibration_check"
    COMPLIANCE_AUDIT = "compliance_audit"
    RISK_ASSESSMENT = "risk_assessment"
    INCIDENT_INVESTIGATION = "incident_investigation"
    MANUAL_REVIEW = "manual_review"
    SPECIFICATION_ANALYSIS = "specification_analysis"
```

### 4. Domain Objects

```python
# Tarea especializada
class BiomedicalTask:
    task_id: str
    task_type: BiomedicalTaskType
    device_id: str
    context: dict

# Evaluación de dispositivo
class DeviceAssessment:
    assessment_id: str
    severity: AssessmentSeverity
    findings: list[dict]
    recommendations: list[str]

# Recomendación de mantenimiento
class MaintenanceRecommendation:
    recommendation_id: str
    maintenance_type: MaintenanceType
    priority: MaintenancePriority
    estimated_cost: float
```

## Consequences

### Positive

1. **Especialización**: Agente enfocado en un dominio
2. **Modularidad**: Expertos son independientes
3. **Testabilidad**: Cada experto testeable aisladamente
4. **Reusabilidad**: Expertos pueden ser reutilizados
5. **Extensibilidad**: Fácil agregar nuevos tipos de tareas

### Negative

1. **Acoplamiento**: Expertos acoplados al agente
2. **Granularidad**: Para tareas simples puede ser overhead
3. **Complejidad**: Más clases que un agente simple

### Neutral

1. **Estado**: Expertos son stateless por diseño
2. **Configuración**: Cada experto configurable

---

## Alternatives Considered

### Alternative 1: Múltiples Agentes Especializados

```
BiomedicalAgent (empty) → EquipmentAgent + MaintenanceAgent + ...
```

**Pros**: Máximo desacoplamiento
**Cons**: Overhead de mensajería, más сложность para Orchestrator

### Alternative 2: Agente Monolítico

```python
class BiomedicalAgent:
    async def analyze_equipment(self, ...): ...
    async def plan_maintenance(self, ...): ...
    async def check_standards(self, ...): ...
```

**Pros**: Simple, sin delegation
**Cons**: Viola SRP, difícil de mantener

### Alternative 3: Plugin Architecture

```python
class Expert(Protocol):
    async def analyze(self, task: Task) -> Result: ...

class BiomedicalAgent:
    def register_plugin(self, plugin: Expert): ...
```

**Pros**: Muy extensible
**Cons**: Más abstracto, overkill para el scope actual

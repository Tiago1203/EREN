# ADR-0053: Agent Orchestrator Architecture

*Status: Proposed*
*Date: 2026-07-23*
*Deciders: Architecture Board*

---

## Context

PHASE 5 requiere un sistema que orqueste múltiples agentes especializados para resolver solicitudes clínicas complejas. El orquestador debe:

- Analizar la solicitud entrante
- Determinar qué agentes son necesarios
- Planificar el orden de ejecución
- Distribuir información a cada agente
- Combinar los resultados de manera coherente
- Garantizar explicabilidad y auditoría

El desafío es diseñar un orquestador que sea:
- **Flexible**: Soporte múltiples estrategias de ejecución
- **Escalable**: Pueda agregar nuevos tipos de agentes
- **Robusto**: Maneje errores y timeouts apropiadamente
- **Eficiente**: Optimice el uso de recursos

---

## Decision

Se adopta una **arquitectura de orquestación basada en componentes** con las siguientes decisiones:

### 1.分离 de Responsabilidades

```
┌────────────────────────────────────────────────────────────────┐
│                    OrchestratorEngine                          │
│  (Coordinación general, recibe solicitudes)                   │
└──────────────────────────┬─────────────────────────────────────┘
                           │
     ┌─────────────────────┼─────────────────────┐
     │                     │                     │
     ▼                     ▼                     ▼
┌─────────┐        ┌─────────────┐        ┌────────────┐
│Dispatcher│        │  Scheduler  │        │ Aggregator │
│         │        │             │        │            │
│ Envía   │        │ Programa    │        │ Combina    │
│ tareas  │        │ ejecución   │        │ resultados │
└─────────┘        └─────────────┘        └────────────┘
```

**Ventajas:**
- Cada componente tiene responsabilidad única
- Fácil de probar individualmente
- Permite reemplazar estrategias sin cambiar otros componentes

### 2. Planificación en Dos Fases

**Fase 1: Creación de Plan**
```
Query → Intent Analyzer → Agent Selector → OrchestrationPlan
```

**Fase 2: Ejecución de Workflow**
```
Plan → Workflow Engine → Step Execution → AggregatedResult
```

**Ventajas:**
- Separación entre planificación y ejecución
- Plan puede ser validado antes de ejecutar
- Permite "dry-run" para debugging

### 3. Estrategias de Ejecución

| Estrategia | Uso | Implementación |
|------------|-----|----------------|
| `SEQUENTIAL` | Pasos con dependencias | Topological sort |
| `PARALLEL` | Pasos independientes | asyncio.gather |
| `PIPELINE` | Procesamiento en cadena | Salida→Entrada |
| `FAN_OUT` | Distribución | TaskGroup |
| `HYBRID` | Combinación | Múltiples fases |

### 4. Modelo de Contexto Compartido

```
Workflow
├── input_context    # Datos iniciales
├── shared_context   # Acumulado entre pasos
└── step_results     # Resultados por paso
```

Cada paso puede:
- Leer del `shared_context`
- Escribir al `shared_context`
- Almacenar su resultado en `step_results`

### 5. Agregación de Resultados

| Método | Selección | Uso |
|--------|-----------|-----|
| `FIRST` | Primer válido | Latencia mínima |
| `BEST` | Mayor confianza | Máxima calidad |
| `ALL` | Todos | Análisis completo |
| `MERGE` | Fusión profunda | Consistencia |
| `VOTE` | Mayoría | Consenso |
| `WEIGHTED` | Ponderado | Confianza vs velocidad |

## Consequences

### Positive

1. **Modularidad**: Componentes débilmente acoplados
2. **Flexibilidad**: Estrategias intercambiables
3. **Testabilidad**: Cada componente testeable aisladamente
4. **Extensibilidad**: Nuevas estrategias sin modificar existentes
5. **Observabilidad**: Cada paso genera eventos y métricas

### Negative

1. **Complejidad**: Más componentes que una solución monolítica
2. **Overhead**: Más abstracciones para casos simples
3. **Latencia**: Cada capa añade latencia mínima
4. **Curva de aprendizaje**: Requiere entender múltiples componentes

### Neutral

1. **Estado**: Necesita gestión de estado distribuido
2. **Configuración**: Más opciones de configuración
3. **Monitoreo**: Más puntos de logging/métricas

## Alternatives Considered

### Alternative 1: Monolito Orchestrator

```python
class SimpleOrchestrator:
    async def process(self, query):
        # Todo en un método
        agents = self.select_agents(query)
        for agent in agents:
            result = await agent.execute(...)
        return self.combine(results)
```

**Pros**: Simple, menos código
**Cons**: Difícil de extender, no separable

### Alternative 2: Event-Driven (Actores)

```python
actor PlanActor:
    def receive(self, message):
        # Procesa mensajes
```

**Pros**: Muy desacoplado
**Cons**: Overhead de mensajes, debugging difícil

### Alternative 3: Graph-based (LangGraph style)

```python
graph = (
    step("biomedical")
    .then("diagnostic")
    .branch(...)
)
```

**Pros**: Declarativo, visualizable
**Cons**: Paradigma nuevo, curva de aprendizaje

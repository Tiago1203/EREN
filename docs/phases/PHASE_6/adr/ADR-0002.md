# ADR-0002: Dashboard Migration Strategy

*Status: Proposed*
*Date: 2026-07-24*
*Deciders: Architecture Board*

---

## Context

Necesitamos migrar el dashboard existente (`apps/web/src/app/(dashboard)/dashboard/`) al nuevo esquema modular manteniendo la funcionalidad actual y agregando capacidades de navegación.

El dashboard actual tiene:
- Stats cards (equipos, mantenimientos, establecimientos)
- KPI Grid
- Información de establecimiento
- Navegación inline

---

## Decision

### Estrategia de Migración

1. **Migración incremental**: Copiar componentes al módulo, no mover
2. **Preservar funcionalidad**: No cambiar el comportamiento actual
3. **Extender con tipos**: Agregar tipado completo
4. **Lazy loading**: Cargar módulos bajo demanda

### Componentes a Migrar

| Componente Original | Nuevo Ubicación | Cambios |
|---------------------|-----------------|---------|
| `page.tsx` | `modules/dashboard/pages/page.tsx` | Agregar tipos, extraer lógica |
| `KpiGrid` | `modules/dashboard/components/KpiGrid.tsx` | Tipado completo |
| Sidebar inline | `modules/navigation/components/Sidebar.tsx` | Modularizar |

## Consequences

### Positive
- Dashboard más mantenible
- Componentes reutilizables
- Tipado completo (TypeScript)
- Lazy loading por módulo

### Negative
- Duplicación temporal de código
- Rutas no cambian (mantener compatibilidad)

### Neutral
- Requiere actualizar imports
- Tests deben actualizarse

---

## Alternatives Considered

### Alternative 1: Mover todo de una vez
- **Rechazado**: Riesgo de romper la aplicación

### Alternative 2: No migrar, dejar como está
- **Rechazado**: Acumula deuda técnica

### Alternative 3: Migración incremental (Elegido)
- **Aceptado**: Bajo riesgo, permite validar cada paso

# ADR-0001: Frontend Architecture Decision

*Status: Approved*
*Date: 2026-07-24*
*Deciders: Architecture Board*

---

## Context

Necesitamos establecer la arquitectura del frontend para PHASE 6 (Hospital Platform) de manera que:
- Soporte crecimiento durante FASE 6 y 7
- No genere deuda técnica
- Permanezca desacoplado del AI Core (FASE 1-5)
- Reutilice la interfaz existente

El frontend debe consumir el AI Core pero no depender de su implementación interna.

---

## Decision

Implementar arquitectura **feature-first modular** con las siguientes características:

1. **Estructura de módulos autocontenidos**
   - Cada módulo tiene: components, hooks, services, api, stores, types, utils, pages
   - Módulos existentes: dashboard, kpis, equipos, mantenimientos, establecimientos
   - Módulos nuevos: ai, analytics, reports, notifications, operations, administration, connectors, knowledge, workspace

2. **Carpeta shared/ para utilidades compartidas**
   - components/: UI base (Button, Input, Card, Modal, Table, Badge)
   - hooks/: Hooks globales (useAuth, useToast, useDebounce)
   - lib/: Configuración (supabase, constants)
   - types/: Tipos globales
   - utils/: Utilidades (format, date)

3. **Rutas Next.js delegan a módulos**
   - app/ contiene solo rutas y layouts
   - Lógica de negocio en modules/
   - Lazy loading por módulo

4. **Feature Flags para módulos no implementados**
   - CONNECTORS preparado pero no habilitado
   - Permite rollout gradual

---

## Consequences

### Positive
- Escalabilidad: nuevos módulos no afectan existentes
- Mantenibilidad: cada módulo es independiente
- Reutilización: shared/ evita duplicación
- Desacoplamiento: frontend separado del AI Core

### Negative
- Curva de aprendizaje: estructura diferente a apps/web original
- Migración: requiere mover código existente

### Neutral
- Requiere configuración inicial de módulos
- Feature flags会增加 complejidad menor

---

## Alternatives Considered

### Alternative 1: Mantener estructura plana
- Todo en components/, hooks/, lib/
- **Rechazado**: No escala, mezcla responsabilidades

### Alternative 2: Módulos por tecnología
- Separar por React, Vue, etc.
- **Rechazado**: No aplica, solo React

### Alternative 3: Feature-first modular (Elegido)
- Módulos autocontenidos por funcionalidad
- **Aceptado**: Escalable, mantenible, prueba de futuro

---

## Implementation Notes

```typescript
// Structure
apps/web/src/
├── app/           # Routes only
├── modules/       # Feature modules
│   ├── dashboard/
│   ├── ai/
│   └── ...
└── shared/        # Shared utilities
    ├── components/
    ├── hooks/
    └── ...
```

- Módulos existentes se migrarán progresivamente
- AI SDK se publicará como paquete npm
- Feature flags configurables por environment

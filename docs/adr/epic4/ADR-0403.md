# ADR-0403: RAG Architecture

**Status:** Proposed

**Date:** 2026-07-20

**Deciders:** Architecture Board

---

## Context

EREN necesita recuperar conocimiento relevante para fundamentar sus respuestas en evidencia. El sistema RAG (Retrieval-Augmented Generation) debe:

1. **Buscar en Knowledge Articles** — Contenido curado de la base de conocimientos
2. **Buscar en Domain Entities** — Devices, Incidents, Staff, etc.
3. **Fusionar resultados** — Combinar evidencia de múltiples fuentes
4. **Citar fuentes** — Proveer referencias claras

Epic 0 ADR-0056 estableció Knowledge Retrieval como capability. Este ADR define la implementación de RAG.

---

## Decision

**Arquitectura RAG de 3 niveles:**

```
┌─────────────────────────────────────────────────────────────┐
│                    RAG Orchestrator                         │
│                                                             │
│  1. Query Understanding → Rewrite, expand, decompose        │
│  2. Multi-source Retrieval → Knowledge + Entities          │
│  3. Fusion → Rerank, deduplicate, combine                  │
│  4. Citation → Link evidence to sources                    │
└─────────────────────────────────────────────────────────────┘
         │                    │                    │
         ▼                    ▼                    ▼
┌─────────────────┐ ┌─────────────────┐ ┌─────────────────┐
│  Knowledge      │ │   Entity       │ │   Hybrid       │
│  Retriever      │ │   Retriever    │ │   Search       │
│  (Qdrant)       │ │   (PostgreSQL) │ │                 │
└─────────────────┘ └─────────────────┘ └─────────────────┘
```

**Retrievers:**

```python
    """Combines vector + keyword search."""

        self,

        1. Vector search (Qdrant)

        """
        ...
```

**RAG Orchestrator:**

```python
class RAGOrchestrator:
    """Coordinates multi-source retrieval."""
    
    def __init__(
        self,
        knowledge_retriever: KnowledgeRetriever,
        entity_retriever: EntityRetriever,
        hybrid_retriever: HybridRetriever,
        reranker: Reranker,
    ):
        ...
    
    async def retrieve(
        self,
        context: RetrievalContext,
    ) -> RetrievalResult:
        """
        1. Understand query (rewrite/decompose if complex)
        2. Parallel retrieval from all sources
        3. Fuse results using RRF
        4. Rerank top-K
        5. Add citations
        6. Return with provenance
        """
        evidence = []
        
        # Parallel retrieval
        knowledge_task = self.knowledge_retriever.retrieve(context.query, context.filters)
        entity_task = self.entity_retriever.retrieve(context.query, context.entity_types, context.filters)
        hybrid_task = self.hybrid_retriever.retrieve(context.query, context.filters)
        
        results = await asyncio.gather(knowledge_task, entity_task, hybrid_task)
        
        # Fusion
        fused = self.rrf_fusion(*results)
        
        # Rerank
        reranked = await self.reranker.rerank(context.query, fused, top_k=10)
        
        # Add citations
        for item in reranked:
            item.citation = self.create_citation(item)
        
        return RetrievalResult(chunks=reranked, total_sources=len(set(r.source_id for r in reranked)))
```

## Reasons

**1. Multi-source retrieval:**
- Knowledge Articles proporcionan evidencia curada
- Domain entities proporcionan contexto operacional
- Hybrid combina ambos enfoques

**2. RRF fusion:**
- Técnica robusta para combinar rankings
- No requiere training
- Handles diferentes score ranges

**3. Reranking:**
- Cross-encoder para mejor relevancia
- Commercial: E5, BGE-Reranker

**4. Citation tracking:**
- Traceability a fuentes originales
- Crucial para compliance médico
- UX: usuario puede verificar evidencia

---

## Consequences

**Positive:**
- Evidence-backed responses
- Reduced hallucination
- Full traceability
- Domain-aware (uses entity data)

**Negative:**
- Additional latency (retrieval step)
- Qdrant dependency
- Embedding model management
- Chunking strategy affects quality

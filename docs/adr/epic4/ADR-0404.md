# ADR-0404: Conversation Model

**Status:** Proposed

**Date:** 2026-07-20

**Deciders:** Architecture Board

---

## Context

EREN necesita manejar conversaciones largas y significativas con usuarios. El modelo de conversación debe:

1. **Mantener contexto** — Historial relevante accesible
2. **Ser escalable** — Miles de conversaciones simultáneas
3. **Ser auditables** — Compliance requirements
4. **Soportar multimodality** — Text, structured data, actions

Epic 0 define Conversation como capability. Este ADR define el modelo de datos y APIs.

---

## Decision

**Modelo de datos:**

```python
# Conversation aggregate
@dataclass
class Conversation:
    id: ConversationId
    tenant_id: TenantId
    user_id: UserId
    session_id: SessionId
    status: ConversationStatus  # ACTIVE, ENDED, ARCHIVED
    created_at: datetime
    updated_at: datetime
    ended_at: datetime | None
    
    # Context
    primary_domain: Domain  # CLINICAL, BIOMEDICAL, HOSPITAL
    context_snapshot: dict  # Current context summary
    
    # Metadata
    message_count: int
    average_confidence: float
    tags: list[str]

# Message entity
@dataclass
class Message:
    id: MessageId
    conversation_id: ConversationId
    role: MessageRole  # USER, ASSISTANT, SYSTEM
    content: MessageContent
    
    # Timestamps
    created_at: datetime
    latency_ms: int  # Time to generate response
    
    # Quality metrics
    confidence: float
    tokens_used: int
    reasoning_steps: int
    
    # References
    citations: list[Citation]
    tool_calls: list[ToolCall]
    
    # Feedback
    user_feedback: Feedback | None

# Conversation Status
enum ConversationStatus:
    ACTIVE = "active"      # Ongoing conversation
    ENDED = "ended"       # User ended or timeout
    ARCHIVED = "archived" # Moved to cold storage
```

**Conversation Controller API:**

```python
class ConversationController(Protocol):
    """Main interface for conversation management."""
    
    async def create_conversation(
        self,
        user_id: UserId,
        tenant_id: TenantId,
        primary_domain: Domain,
    ) -> Conversation:
        """Creates a new conversation session."""
        ...
    
    async def send_message(
        self,
        conversation_id: ConversationId,
        message: Message,
    ) -> Response:
        """Sends a message and gets AI response."""
        ...
    
    async def send_message_stream(
        self,
        conversation_id: ConversationId,
        message: Message,
    ) -> AsyncGenerator[ResponseChunk]:
        """Sends message with streaming response."""
        ...
    
    async def get_conversation(
        self,
        conversation_id: ConversationId,
    ) -> Conversation:
        """Gets conversation details."""
        ...
    
    async def get_history(
        self,
        conversation_id: ConversationId,
        limit: int = 50,
        before: datetime | None = None,
    ) -> list[Message]:
        """Gets message history with pagination."""
        ...
    
    async def end_conversation(
        self,
        conversation_id: ConversationId,
        reason: EndReason,
    ) -> Conversation:
        """Ends a conversation gracefully."""
        ...
    
    async def archive_conversation(
        self,
        conversation_id: ConversationId,
    ) -> None:
        """Archives conversation to cold storage."""
        ...
```

**Session Management:**

```python
@dataclass
class Session:
    id: SessionId
    conversation_id: ConversationId
    user_id: UserId
    
    # Session context (Redis)
    working_memory: dict  # Current context
    recent_messages: list[MessageSummary]  # Last N for context window
    
    # Session config
    temperature: float
    max_tokens: int
    streaming_enabled: bool
    
    # TTL: 30 minutes of inactivity
    created_at: datetime
    last_activity: datetime
```

---

## Reasons

**1. Scalability:**
- Sessions en Redis (in-memory)
- Archived conversations en PostgreSQL
- Hot/warm/cold storage strategy

**2. Auditability:**
- Every message logged
- Token usage tracked
- User feedback stored
- Compliance-ready

**3. UX:**
- Streaming para real-time feedback
- Pagination para history
- Context snapshots para quick resume

**4. Cost control:**
- Token counting por message
- Budget tracking por conversation
- Alerting on high usage

---

## Consequences

**Positive:**
- Scalable to thousands of concurrent users
- Full audit trail
- Cost tracking per conversation
- Good UX with streaming

**Negative:**
- More complex than stateless API
- Redis dependency for sessions
- Storage costs for long conversations

# ADR-0310: Work Order Integration with Hospital

**Status:** Proposed

**Date:** 2026-07-16

**Deciders:** Architecture Board

---

## Context

Work Orders (from EPIC 2, Incident Context) must integrate with the Hospital domain contexts. A Work Order is triggered by an Incident, but its resolution requires information from Hospital contexts: the asset location (Capacity), assigned staff (Staffing), spare parts (Inventory), and maintenance records (Asset).

---

## Decision

**Integration points:**

```
Incident Context (EPIC 2)
└── EngineeringIncident aggregate
    └── work_order_id (reference — not FK, to preserve context isolation)
        │
        ├── Capacity Context ← asset location (room_id → building → campus)
        ├── Staffing Context ← assigned biomedical engineer (staff_id)
        ├── Inventory Context ← spare parts used (purchase_order_id)
        ├── Asset Context    ← asset history, warranty status
        └── Maintenance Context ← preventive maintenance records
```

**Integration pattern: Events (no direct FK between bounded contexts)**

```
EngineeringIncidentResolved (from Incident Context)
  → triggers → CorrectiveMaintenancePlanCreated (Maintenance Context)

AssetFailureDetected (from Incident Context)
  → triggers → PreventiveMaintenanceRecommended (Maintenance Context)
  → triggers → SparePartReserved (Inventory Context)

WorkOrderCompleted (from Incident Context)
  → triggers → MaintenanceRecordCreated (Maintenance Context)
  → triggers → SparePartConsumed (Inventory Context)
  → triggers → AssetUpdated (Asset Context — update MTBF)
```

**Work Order entity (from EPIC 2):**

```
WorkOrder (from Incident Context — core/incident/domain/work_order/)
├── tenant_id, work_order_id, incident_id (FK)
├── asset_id (FK → Asset Context)
├── assigned_staff_id (FK → Staffing Context)
├── location_id (FK → Capacity Context: room_id)
├── priority: enum
├── status: enum
├── scheduled_date: date
├── estimated_hours: decimal
├── actual_hours: decimal (nullable)
├── parts_used: list[PartUsed] (value object)
├── completion_notes: string
└── sla_status: enum
```

**Key integration rules:**
- `asset_id` links to Asset Context (where the asset was registered)
- `location_id` links to Capacity Context (where the asset is physically located)
- `assigned_staff_id` links to Staffing Context (who performs the work)
- `parts_used` references SparePart IDs from Inventory Context

---

## Consequences

**Positive:**
- Bounded context isolation preserved via events (no circular FK dependencies)
- Each context owns its data; integration is via domain events
- Audit trail via event sourcing

**Negative:**
- Eventual consistency between contexts (events may have small delay)
- Cross-context queries require event aggregation

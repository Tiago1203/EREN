# ADR-0301: Capacity Bounded Context

**Status:** Proposed

**Date:** 2026-07-16

**Deciders:** Architecture Board

---

## Context

The Capacity Context manages the physical hospital structure: beds, rooms, floors, buildings, and campuses. It tracks occupancy and enables capacity planning. It is the primary context for bed management — a critical operational need in hospitals.

---

## Decision

**Aggregate: Bed** (aggregate root)

```
Bed (aggregate root)
├── tenant_id, campus_id, building_id, floor_id, room_id, bed_id
├── bed_number: string
├── bed_type: enum (standard, ICU, pediatric, neonatal, maternity, isolation)
├── status: enum (available, occupied, blocked, maintenance, closed)
├── floor_id (FK)
└── room_id (FK)
```

**Sub-entities:**

```
Room
├── tenant_id, campus_id, building_id, floor_id, room_id
├── room_number: string
├── room_type: enum (patient, OR, procedure, storage, utility)
├── bed_count: integer
└── status: enum (operational, maintenance, closed)

Floor
├── tenant_id, campus_id, building_id, floor_id
├── floor_number: integer
├── floor_type: enum (basement, ground, standard, rooftop)
└── status: enum (operational, under_maintenance, closed)

Building
├── tenant_id, campus_id, building_id
├── building_code: string
├── building_name: string
├── building_type: enum (main, pavilion, annex, parking)
└── status: enum (operational, under_maintenance, closed)

Campus
├── tenant_id, campus_id, organization_id
├── campus_code: string
├── campus_name: string
├── address: Address (value object)
├── geographic_coordinates: GeoPoint (value object)
└── status: enum (active, inactive, planning)
```

**Invariants:**
- A bed must always belong to exactly one room
- A room must always belong to exactly one floor
- A floor must always belong to exactly one building
- A building must always belong to exactly one campus
- A campus must always belong to exactly one organization
- Bed status can only transition: available ↔ occupied ↔ blocked ↔ maintenance ↔ closed
- Isolation beds require negative pressure flag

**Domain Events:**
- `BedCreated`, `BedOccupied`, `BedVacated`, `BedBlocked`, `BedUnblocked`
- `RoomConfigured`, `RoomStatusChanged`, `FloorConfigured`, `BuildingRegistered`, `CampusRegistered`
- `UnitCapacityChanged`, `CapacityAlertTriggered`

---

## Consequences

**Positive:**
- Bed is the aggregate root — all capacity decisions centered on bed state
- Clear hierarchy: Campus → Building → Floor → Room → Bed
- Capacity alerts triggered on threshold crossings
- Multi-campus support built-in

**Negative:**
- Hierarchy traversal requires multiple joins
- Capacity calculations need aggregation across campus

# ADR-0308: Organization Hierarchy Model

**Status:** Accepted

**Date:** 2026-07-16

**Deciders:** Architecture Board

---

## Context

This ADR details the Organization hierarchy implementation, complementing ADR-0051 (Organization Multi-Campus Strategy) from Epic 0.

The Organization hierarchy defines the legal and administrative structure of a hospital organization.

---

## Decision

**Organization Hierarchy:**

```
Organization (legal entity)
├── Tenant (registry — from EPIC 1)
└── Organization
    ├── Hospital (operational unit)
    │   ├── Campus (geographic cluster)
    │   │   ├── Building
    │   │   │   ├── Floor
    │   │   │   │   └── Room
    │   │   │   │       └── Bed
```

**Key rules:**

1. **One Tenant = One Organization:** A tenant_id always maps to one legal Organization
2. **Multi-Campus:** One Organization can have multiple Campuses (all share tenant_id)
3. **Campus isolation:** Each Campus has a unique campus_id; all physical entities include campus_id
4. **Hierarchical IDs:** Physical entity = tenant_id + campus_id + building_id + floor_id + room_id + bed_id

**Multi-tenant enforcement:**
- All Organization, Campus, Building, Floor, Room, Bed tables have tenant_id
- RLS policies restrict cross-tenant access (from EPIC 1 migration 007)
- API middleware enforces tenant_id injection (from EPIC 1)

**Data model:**

```sql
hospital.organization (
    tenant_id UUID PK,
    organization_id UUID PK,
    legal_name TEXT,
    tax_id TEXT,
    ownership_type TEXT,
    ...
)

hospital.campus (
    tenant_id UUID PK,
    campus_id UUID PK,
    organization_id UUID FK,
    campus_code TEXT,
    ...
)

hospital.building (
    tenant_id UUID PK,
    campus_id UUID PK,
    building_id UUID PK,
    ...
    FOREIGN KEY (tenant_id, campus_id) REFERENCES hospital.campus
)

hospital.floor (
    tenant_id UUID PK,
    campus_id UUID PK,
    building_id UUID PK,
    floor_id UUID PK,
    ...
    FOREIGN KEY (tenant_id, campus_id, building_id) REFERENCES hospital.building
)

hospital.room (
    tenant_id UUID PK,
    campus_id UUID PK,
    building_id UUID PK,
    floor_id UUID PK,
    room_id UUID PK,
    ...
    FOREIGN KEY (tenant_id, campus_id, building_id, floor_id) REFERENCES hospital.floor
)

hospital.bed (
    tenant_id UUID PK,
    campus_id UUID PK,
    building_id UUID PK,
    floor_id UUID PK,
    room_id UUID PK,
    bed_id UUID PK,
    ...
    FOREIGN KEY (tenant_id, campus_id, building_id, floor_id, room_id) REFERENCES hospital.room
)
```

---

## Consequences

**Positive:**
- Hierarchical IDs enable efficient tree traversal queries
- Multi-campus organizations fully supported
- Clear tenant isolation at all levels
- Compatible with PostGIS for geographic queries (ADR-0300)

**Negative:**
- Composite FKs (5 columns) for bed → organization path
- Renaming a campus_id requires cascade updates

# ADR-0094: Schema Registry Strategy

**Status:** Proposed

**Date:** 2026-07-16

**Deciders:** Infrastructure Team

---

## Context

EREN publishes events to Kafka. Without schema management, event schema changes can break consumers silently. Schema registry ensures backward compatibility.

---

## Decision

**Confluent Schema Registry is used for Kafka event schema management.**

---

## Schema Strategy

| Aspect | Decision |
|--------|---------|
| Format | JSON Schema (not Avro — simpler, wider tool support) |
| Compatibility | BACKWARD only — new fields must have defaults |
| Validation | Schema validated on produce and consume |
| Versioning | Schema versions stored in registry |

---

## Schema Registry Setup

```python
# Producer: validate before publishing
schema = schema_registry.get_schema("eren-clinical.patients.created")
validated_payload = schema.validate(event.payload)

# Consumer: validate on receive
schema = schema_registry.get_schema(event.headers["schema_version"])
validated_payload = schema.validate(event.payload)
```

---

## Schema Evolution Rules

| Change | Compatible |
|--------|-----------|
| Add new field with default | ✅ Yes |
| Remove field | ✅ Yes |
| Rename field | ❌ No (use alias pattern) |
| Change field type | ❌ No |
| Add required field | ❌ No |

---

## Topic Schema Mapping

Each Kafka topic has a registered schema:
```
eren-clinical.patients.created  → PatientCreated schema
eren-clinical.diagnosis.events → Diagnosis schema
eren-biomedical.device.events   → DeviceEvent schema
eren-biomedical.alarm.events   → Alarm schema
```

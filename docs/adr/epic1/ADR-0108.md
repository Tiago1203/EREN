# ADR-0108: RabbitMQ Messaging

**Status:** Accepted

**Date:** 2026-07-16

**Deciders:** Architecture Board

---

## Context

EREN uses event-driven communication between bounded contexts. A message broker is needed to decouple producers from consumers and enable async processing of domain events.

## Decision

Use **RabbitMQ 3.13+** with **aio-pika** as the async client and a **topic exchange** named `eren.events`.

## Rationale

- **AMQP**: Industry standard for async messaging
- **Topic exchange**: Supports routing patterns like `device.#`, `incident.created`
- **Durable queues**: Messages survive broker restarts
- **Management UI**: Built-in web UI at port 15672
- **aio-pika**: Full async support, integrates with FastAPI

## Exchange Design

| Exchange | Type | Durable | Purpose |
|---------|------|---------|---------|
| `eren.events` | topic | yes | Domain events from all contexts |
| `eren.commands` | direct | yes | Command messages |
| `eren.dlq` | fanout | yes | Dead letter queue |

### Topic Routing Patterns

```python
# Publish
topic = "device.alert.critical"
payload = {"device_id": "...", "severity": "critical"}
await event_bus.publish(topic, payload)

# Subscribe
queue = await channel.declare_queue("engineering-alerts")
await queue.bind(exchange, routing_key="device.alert.*")
```

## Consequences

### Positive

- Full decoupling between bounded contexts
- Async processing of events via consumer workers
- DLQ for failed message handling
- Management UI for monitoring

### Negative

- Operational complexity (RabbitMQ must be monitored)
- Message ordering not guaranteed across partitions
- At-least-once delivery (outbox pattern mitigates this)

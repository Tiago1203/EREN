# ADR-0103: Repository Pattern Implementation

**Status:** Accepted

**Date:** 2026-07-16

**Deciders:** Architecture Board

---

## Context

The bounded contexts in Epic 0 define clear boundaries. Each context must expose a repository interface that isolates the domain from the database technology.

## Decision

Implement the **Repository pattern** with SQLAlchemy as the implementation. Each bounded context has one repository interface in `core/{context}/domain/repositories/` and one implementation in `apps/api/app/infrastructure/repositories/`.

## Rationale

- **Bounded context isolation**: Each context owns its repository, no cross-context DB queries
- **Testability**: Repositories can be mocked in unit tests
- **Swappable storage**: PostgreSQL can be replaced without touching domain logic
- **Clean separation**: Domain entities have no SQLAlchemy imports

## Repository Structure

```
core/
  device/domain/repositories/
    device_repository.py      ← Abstract interface (Protocol)

apps/api/app/infrastructure/repositories/
  device.py                   ← SQLAlchemy implementation
```

## Contract

```python
from typing import Protocol
from core.shared import DeviceId, TenantId, Result

class DeviceRepository(Protocol):
    async def save(self, device: Device) -> Result[Device, str]: ...
    async def get_by_id(self, device_id: DeviceId) -> Result[Device | None, str]: ...
    async def get_by_tenant(
        self, tenant_id: TenantId, limit: int = 100, offset: int = 0
    ) -> Result[list[Device], str]: ...
```

## Bounded Contexts with Repositories

| Bounded Context | Interface | Implementation |
|----------------|-----------|----------------|
| Device | `core.device.domain.repositories.DeviceRepository` | `app.infrastructure.repositories.device.DeviceRepositoryImpl` |
| Incident | `core.incident.domain.repositories.IncidentRepository` | `app.infrastructure.repositories.incident.IncidentRepositoryImpl` |
| Recommendation | `core.recommendation.domain.repositories.RecommendationRepository` | `app.infrastructure.repositories.recommendation.RecommendationRepositoryImpl` |
| Knowledge | `core.knowledge.domain.repositories.KnowledgeRepository` | `app.infrastructure.repositories.knowledge.KnowledgeRepositoryImpl` |

## Consequences

### Positive

- Domain logic never references SQLAlchemy
- Easy to mock for unit tests
- Clear ownership of each repository by its bounded context

### Negative

- Extra abstraction layer adds indirection
- Two files to update when changing data access patterns

# ADR-0107: Redis Cache Strategy

**Status:** Accepted

**Date:** 2026-07-16

**Deciders:** Architecture Board

---

## Context

EREN needs a caching layer to reduce database load, improve response times for read-heavy operations (device lookups, knowledge retrieval), and enable session/token storage.

## Decision

Use **Redis 7+** with a **cache-aside** pattern for read-heavy operations and **TTL-based expiration**.

## Rationale

- **In-memory**: Sub-millisecond response times
- **TTL support**: Automatic expiration reduces stale data risk
- **Async**: `redis.asyncio` supports non-blocking operations
- **Pub/Sub**: Can be used for real-time notifications in future EPICs
- **Persistence**: RDB snapshots for development durability

## Cache Strategy

| Operation | Strategy | TTL |
|-----------|----------|-----|
| Device by ID | Cache-aside | 5 min |
| Device list (by tenant) | Cache-aside | 2 min |
| Knowledge article | Cache-aside | 30 min |
| Recommendation | No cache | — |
| Session/token | Write-through | 1 hour |
| Rate limiting | Counter | 1 min |

## Usage

```python
# apps/api/app/infrastructure/messaging/cache.py
from redis.asyncio import Redis

async def get_device_cache(device_id: str) -> Device | None:
    client = await get_redis()
    if client is None:
        return None

    cached = await client.get(f"device:{device_id}")
    if cached:
        return Device.model_validate_json(cached)

    return None

async def set_device_cache(device: Device, ttl: int = 300) -> None:
    client = await get_redis()
    if client:
        await client.setex(
            f"device:{device.id}",
            ttl,
            device.model_dump_json(),
        )
```

## Consequences

### Positive

- Significantly reduces PostgreSQL load for repeated queries
- Sub-5ms response for cached reads
- Graceful degradation (returns None if Redis unavailable)

### Negative

- Cache invalidation complexity
- Stale data risk if TTL is too long
- Redis unavailability requires fallback to DB

# ADR-0105: Outbox Pattern for Event Publishing

**Status:** Accepted

**Date:** 2026-07-16

**Deciders:** Architecture Board

---

## Context

EREN uses event-driven communication between bounded contexts (see Epic 0 Event Architecture). Publishing events to RabbitMQ directly from the API introduces the risk of "committed data but missing events" if RabbitMQ is unavailable at commit time.

## Decision

Implement the **Transactional Outbox pattern**: events are written to an `outbox_events` table in the same database transaction as the domain data. A background `OutboxWorker` polls the outbox and publishes to RabbitMQ.

## Rationale

- **No distributed transactions**: Uses single PostgreSQL transaction instead of 2PC
- **At-least-once delivery**: OutboxWorker retries until acknowledgment
- **Event sourcing foundation**: Events are durable and queryable
- **Zero event loss**: Data and events commit or roll back together

## Architecture

```
API Request
    ↓
UnitOfWork.commit()
    ↓
┌─────────────────────────────────────────┐
│ PostgreSQL (single transaction)           │
│   • device INSERT                        │
│   • incident INSERT                      │
│   • outbox_events INSERT (pending)        │
└─────────────────────────────────────────┘
    ↓
OutboxWorker (separate process)
    ↓
┌─────────────────────────────────────────┐
│ PostgreSQL (SELECT FOR UPDATE SKIP LOCKED)│
│   • Fetch pending outbox_events          │
│   • Publish to RabbitMQ                  │
│   • UPDATE status = published            │
└─────────────────────────────────────────┘
```

## Table Schema

```sql
CREATE TABLE incident.outbox_events (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    aggregate_type VARCHAR(255) NOT NULL,
    event_type VARCHAR(255) NOT NULL,
    payload JSONB NOT NULL,
    correlation_id VARCHAR(255),
    status VARCHAR(20) NOT NULL DEFAULT 'pending',
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    processed_at TIMESTAMPTZ,
    retry_count INT NOT NULL DEFAULT 0,
    last_error TEXT,
    CONSTRAINT chk_status CHECK (status IN ('pending', 'published', 'failed'))
);
```

## Consequences

### Positive

- Events and data are always in sync (same transaction)
- RabbitMQ downtime does not block API commits
- DLQ support via retry_count and last_error columns

### Negative

- Events are not published immediately (eventual consistency)
- OutboxWorker must be running for events to reach RabbitMQ
- Outbox table grows indefinitely (purge via `processed_at`)

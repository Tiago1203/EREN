# ADR-0002: PostgreSQL as Primary Database

**Status:** Accepted

**Date:** 2026-01-15

**Deciders:** Architecture Board

---

## Context

EREN needs a primary relational database for all business entities. The system must:
- Store patient records, device data, incidents, and audit logs
- Support multi-tenancy with row-level security
- Handle high write throughput (device telemetry, alarms)
- Support complex queries (patient history, device analytics)
- Be HIPAA-compliant with audit trails

We evaluated PostgreSQL and MySQL.

---

## Decision

**PostgreSQL 16 is the primary database for all bounded contexts.**

All business entities are stored in PostgreSQL. It is the **single source of truth** for transactional data.

---

## Reasons

1. **Row-Level Security (RLS):** Native multi-tenant isolation at the database level — critical for HIPAA compliance
2. **JSONB:** Flexible schema for event payloads without sacrificing queryability
3. **PostGIS:** Geographic queries for hospital campus/building/room (future EPIC 3)
4. **pg_stat_statements:** Built-in query performance monitoring
5. **COPY command:** High-speed bulk imports for device telemetry
6. **Transactional DDL:** Schema migrations are transactional
7. **Epic 0 alignment:** `EREN_CONSISTENCY_MODEL.md` specifies PostgreSQL as SoT

---

## Consequences

### Positive
- Native RLS for multi-tenancy enforcement
- Rich indexing (partial, composite, GIN, GiST)
- JSONB for semi-structured event data
- Strong consistency (ACID transactions)
- Well-understood operational model

### Negative
- Horizontal sharding is complex
- Full-text search less powerful than dedicated search engines
- Schema migrations require careful planning

---

## Data Classification in PostgreSQL

| Classification | Stored In | Encryption |
|---------------|-----------|-----------|
| PHI (patients, diagnoses) | PostgreSQL | At-rest (AES) + RLS |
| Device data | PostgreSQL | At-rest |
| Audit logs | PostgreSQL | Immutable, at-rest |
| Configuration | PostgreSQL | No PHI |

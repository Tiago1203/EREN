# ADR-0004: Redis Cache-Aside Pattern

**Status:** Accepted

**Date:** 2026-02-01

**Deciders:** Architecture Board

---

## Context

EREN needs caching for frequently accessed data to reduce PostgreSQL load. The system must cache patient summaries, device configurations, and knowledge base results.

---

## Decision

**Redis is used as a cache with the Cache-Aside pattern.**

Data is cached on read. The cache is invalidated on write. Redis is NEVER the source of truth.

---

## Reasons

1. **Cache-Aside:** Simple, explicit caching — no dual-write complexity
2. **TTL-based expiration:** Automatic staleness prevention
3. **Redis Streams:** Redis 7 streams for Celery task queue backend
4. **Pub/Sub:** Real-time alarm notifications
5. **Rate limiting:** Sliding window rate limiting using Redis

---

## Use Cases

| Use Case | Key Pattern | TTL | Notes |
|----------|-------------|-----|-------|
| Patient summary | `patient:{id}:summary` | 5 min | Cache-aside |
| Device config | `device:{id}:config` | 15 min | Cache-aside |
| CDS evidence | `evidence:{id}` | 30 min | Cache-aside |
| Session | `session:{user_id}` | 24h | JWT refresh tokens |
| Rate limit | `ratelimit:{tenant}:{endpoint}` | 60s | Sliding window |
| Celery broker | — | — | Redis Streams (see ADR-0093) |

---

## Consequences

### Positive
- Reduces PostgreSQL load significantly
- Sub-millisecond access for cached data
- Multiple use cases (cache, session, rate limiting, queue)

### Negative
- Cache invalidation complexity
- Potential cache stampede on cold start
- Redis failure requires fallback to PostgreSQL

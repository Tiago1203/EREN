# ADR-0005: Neo4j for Knowledge Graph

**Status:** Accepted

**Date:** 2026-02-15

**Deciders:** Architecture Board

---

## Context

EREN needs to model complex relationships: device-to-device dependencies, patient-device associations, alarm propagation chains, and root cause causal graphs. A relational database would require complex JOINs and self-referential tables.

---

## Decision

**Neo4j 5 is used for relationship modeling and graph queries.**

PostgreSQL remains the source of truth. Neo4j is a **derived store** that receives updates via domain events.

---

## Reasons

1. **Graph-native queries:** Cypher is natural for relationship traversal
2. **Alarm propagation:** Root cause analysis traverses alarm causal chains efficiently
3. **Patient-device associations:** Complex multi-hop queries
4. **Epic 0 alignment:** `EREN_CONSISTENCY_MODEL.md` specifies Neo4j as derived graph store

---

## Use Cases

| Use Case | Example Query |
|----------|-------------|
| Device dependencies | "Find all devices affected if Device X fails" |
| Alarm root cause | "Trace alarm propagation chain back to source" |
| Patient-device links | "Find all devices associated with this patient" |
| Similar incidents | "Find incidents with similar device+symptom patterns" |

---

## Consequences

### Positive
- Natural modeling of complex relationships
- Efficient graph traversal (O(1) per hop vs O(n) for JOINs)
- Visual exploration of relationships

### Negative
- Additional infrastructure to maintain
- Neo4j is derived — must be rebuilt from PostgreSQL on failure
- Training cost for Cypher queries
